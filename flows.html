Ralph Skill Workflow

How /ui-loop, /ralph, and /ralph-brother relate to each other

/ui-loop  ·  UI coverage scan (Phase 1 + 2)
📋
Read prd.json
Loads all requirements. Counts already-mapped vs remaining.
🔄
For each requirement → /ui-check
Spawns one agent per requirement in parallel batches. Each agent scans the React frontend for the requirement's acceptance criteria.
ui-check verdict per requirement
UI_MAPPED — all ACs visible PARTIAL_UI — some ACs missing NO_UI — no frontend surface
💾
Write UI flow report per requirement
Writes ralph/ui/[domain]/PD-XXX.md — plain-English steps + per-AC verdict. Updates prd.json with ui_mapped, ui_partial, ui_no_frontend.
🔍
Phase 2 — Orphan audit
Cross-references every route in App.jsx against all reports. Flags any page with no requirement backing it.
📊
Write ui-coverage-summary.md
Full coverage table (19 mapped / 31 partial / 2 no-UI) plus orphan audit results.

outputs ui_partial / ui_no_frontend flags in prd.json → consumed by /ralph-brother

/ui-check  ·  Single-requirement UI scan (called by /ui-loop)
🗺️
Load nav map
Reads App.jsx (all routes) and Layout.jsx (sidebar tree) to know exactly how users navigate the app.
🔎
For each AC — find the page
Greps src/pages/ with keywords from the criterion. Reads matched files to find buttons, forms, table columns, API calls.
🧭
Trace navigation path
Writes exact click-by-click steps from the top nav bar to the feature. Bolding every element the user clicks.
AC verdict
✅ VISIBLE ⚠️ PARTIAL ❌ NOT VISIBLE
📝
Write UI flow report
One MD file per requirement. Final line = UI_MAPPED / PARTIAL_UI / NO_UI. Read by /ui-loop to update prd.json and later by /ralph-brother Step 2.
/ralph  ·  Original code-gap loop
📋
Queue: requirements where passes: false
Ignores UI status. Focuses only on whether backend + frontend code satisfies each AC at the code level.
🔬
Step 1 — /gap-check
Scans Java + React source. Greps AC keywords. Reads matched files. Writes ralph/gaps/PD-XXX.md with IMPLEMENTED / PARTIAL / MISSING verdict per AC.
GAP_CHECK_RESULT
PASSES → mark done, next req GAPS_FOUND → Step 2
🔧
Step 2 — /code-fix
Reads gap report. Implements every fix (entity, service, controller, DTO, React component, API call). Sets passes: true in prd.json when done.
↩️
Loop back to Step 1
Re-runs gap-check after each fix. Repeats until PASSES or max iterations hit.

writes ralph/gaps/PD-XXX.md → also read by /ralph-brother in its merge step

/ralph-brother  ·  UI-gap fix loop (Option B merge)
📋
Queue: ui_partial OR ui_no_frontend OR passes:false
Processes anything with UI gaps or lingering code gaps. Skips only requirements with ui_mapped: true AND passes: true.
🔬
Step 1 — /gap-check (always runs first)
Finds backend/frontend code gaps independently. Catches missing endpoints and entities before the UI layer is considered. Writes ralph/gaps/PD-XXX.md.
🔀
Step 2 — UI synthesis + merge
Reads ralph/ui/[domain]/PD-XXX.md. For each PARTIAL / NOT VISIBLE AC:
 • Backend gap found by gap-check → upgrade to Layer: Both
 • No gap + no endpoint exists → add new Layer: Both gap
 • No gap + endpoint exists → add Layer: Frontend gap
Rewrites the merged gap report in-place. /code-fix now has full context.
🔧
Step 3 — /code-fix
Reads merged gap report. Implements BE first (entities, endpoints, DTOs), then FE (components, API calls, route wiring). Both layers in one pass.
✅
Step 4 — /ui-check verification
Re-scans the frontend. Only marks done when UI_MAPPED. If still partial → loops back to Step 1 for another iteration (up to max).
UI_CHECK_RESULT
UI_MAPPED → done, update prd.json PARTIAL_UI → loop back max iters → needs_review

Files produced by each skill

ralph/prd.json
Source of truth. All skills read and write flags here: passes, ui_mapped, ui_partial, ui_no_frontend, needs_review.
ralph/ui/[domain]/PD-XXX.md
UI flow report per requirement. Plain-English steps + per-AC verdict. Written by /ui-check, read by /ralph-brother Step 2.
ralph/ui/ui-coverage-summary.md
Full coverage table + orphan audit. Written by /ui-loop after all ui-checks complete.
ralph/ui/ui-loop.log
Append-only event log. Written by /ui-loop and /ralph-brother.
ralph/gaps/PD-XXX.md
Code gap report. Written by /gap-check. Merged with UI gaps and rewritten by /ralph-brother Step 2.
ralph/loop.log
Append-only event log for /ralph original loop.
ralph/loop-summary.md
Final pass/fail summary written by /ralph after all requirements processed.
ralph/ui/brother-summary.md
Final UI fix summary written by /ralph-brother after all requirements processed.

Recommended run order

/ui-loop →
run in parallel
/ralph + /ralph-brother
→ /ui-loop (re-run to confirm)

/ralph fixes code-level gaps (missing endpoints, entities, business logic). It does not care about UI.
/ralph-brother fixes UI-level gaps using the ui-check reports as its specification, with a backend pre-flight via /gap-check to catch missing endpoints before the FE tries to call them.
They can run in parallel — ralph targets ralph/gaps/ exclusively, brother reads those same files only after its own gap-check step writes them fresh.
