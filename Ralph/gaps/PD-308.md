## Frontend UI Gaps

---

### Gap 1 â€” AC3: Language code field is a free-text input rather than a constrained dropdown

**Layer:** Frontend

**AC:** AC3 â€” A language code can be defined for a customer account (Finnish, Swedish, or English)

**What is missing:** The "Language Code" field on the Billing Profile page (`BillingProfilePage.jsx`) is a plain text input. A user can type any arbitrary string (e.g. "de", "fr", "english") and the form will save it without validation. The backend `InvoiceLanguage` enum only supports `FI`, `SV`, and `EN`; an invalid code silently falls back to the Finnish default. The UI should use a constrained dropdown listing "Finnish (fi)", "Swedish (sv)", and "English (en)" so that users cannot enter unsupported codes and understand exactly which options are valid.

### Gap 2 â€” AC5: No UI for defining or viewing product unit translations

**Layer:** Frontend

**AC:** AC5 â€” Translations are defined for product units

**What is missing:** While the backend `PricingUnit` enum includes locale-specific labels and `FinvoiceBuilderService` uses them correctly, there is no screen in the frontend where a user can view, verify, or manage the per-locale unit labels. The Master Data â†’ Products page provides a translations panel for product names (fi/sv/en) but does not display or allow editing of unit label translations. A user has no way to confirm from the UI what unit label (e.g. "kpl", "st", "pcs") will appear on a Finnish, Swedish, or English invoice. A read-only display of unit translations â€” even just a reference table â€” would make this backend behaviour visible and verifiable without requiring a code lookup.
