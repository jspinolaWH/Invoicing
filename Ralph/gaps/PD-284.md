## UI Gaps

### Gap: AC2 — No explicit combine-vs-separate toggle on Billing Profile

**Layer:** Frontend
**AC:** AC2 — Users can configure how public and private transactions are invoiced
**What is missing:** The Billing Profile page has a Default Legal Classification dropdown but no explicit "Combine public and private on one invoice" vs. "Separate into different invoices" toggle. The bundling behavior is only configurable via the BundlingRulesPage as a separate configuration area. Users expect a clear, per-customer invoice grouping preference directly on the billing profile, but must use invoice run filters as a workaround for ad-hoc separation.
**UI surface needed:** A "Invoice Grouping" control (dropdown or radio buttons: COMBINE / SEPARATE) on the Billing Profile page that persists the customer's default bundling preference, or at minimum a clear link from the Billing Profile to the BundlingRules configuration that applies to this customer.

---

### Gap: AC4 — No single "auto-separate by classification" setting on invoice runs

**Layer:** Frontend
**AC:** AC4 — Separate public and private transactions into different invoices
**What is missing:** Separating public and private transactions into different invoices requires the user to manually run two separate invoice runs, each filtered by Service Responsibility (Public Law / Private Law). There is no single configuration toggle that automatically splits an invoice run by legal classification. The manual two-run workaround is not discoverable and error-prone.
**UI surface needed:** A "Split by legal classification" checkbox or toggle on the Invoice Run configuration page that, when enabled, automatically generates separate invoice groups for PUBLIC_LAW and PRIVATE_LAW events in a single run.

---

### Gap: AC5 — No company-level invoice grouping strategy setting

**Layer:** Frontend
**AC:** AC5 — Customize invoice processing rules at the company level
**What is missing:** Classification rules can be configured company-wide (CustomerType, ProductCode, Region → classification result), but there is no company-level setting to define the invoice grouping strategy (whether invoices should combine or separate public/private law charges). The BundlingRulesPage controls bundling per customer/product group, but there is no global company default for this behavior.
**UI surface needed:** A company-level "Default Invoice Grouping" setting (on a company settings or global configuration page) that defines whether new customers default to COMBINE or SEPARATE bundling, reducing per-customer configuration overhead.

---

### Gap: AC7 — Invoice grouping preference is not persisted per customer

**Layer:** Frontend
**AC:** AC7 — User-defined invoice grouping (same or separate invoices)
**What is missing:** The decision of whether to combine or separate public and private transactions on invoices is made ad hoc at each invoice run via the Service Responsibility filter. There is no stored per-customer or per-company preference for invoice grouping by legal classification. Users must remember and re-apply the grouping decision at every invoice run.
**UI surface needed:** A persistent "Preferred Invoice Grouping" field on the Billing Profile or BundlingRules page that stores the customer's grouping preference and pre-fills the invoice run filter accordingly.

---

### Gap: AC9 — No UI surface confirming automatic classification compliance

**Layer:** Frontend
**AC:** AC9 — Automatic compliance with legislation in collection processes
**What is missing:** The system automatically applies classification rules to billing events (PUBLIC_LAW / PRIVATE_LAW), but there is no UI surface that shows the user which rule was matched for a given event, or confirms that the automatic classification is legally compliant. The classification value is visible on the event detail, but the reasoning (which rule matched, what tariff type was identified) is not exposed.
**UI surface needed:** A "Classification Reason" or "Matched Rule" read-only field on the Billing Event Detail page showing which classification rule was applied and why (e.g. "Matched rule: MUNICIPALITY → PUBLIC_LAW, priority 1"). This gives users confidence that the statutory classification logic ran correctly.

---

### Gap: AC10 — No UI to preview or verify FINVOICE classification field mapping

**Layer:** Frontend
**AC:** AC10 — Integration with FINVOICE and external systems (classification exported)
**What is missing:** The FINVOICE export includes classification metadata automatically, but there is no page where users can preview, inspect, or verify that the classification field is correctly mapped in the FINVOICE envelope. Users can download the XML file from the invoice detail attachments panel, but there is no in-app confirmation or preview of the ServiceResponsibility field values in the FINVOICE output.
**UI surface needed:** A FINVOICE preview panel or "FINVOICE export summary" section on the Invoice Detail page that shows the classification value that will be (or was) embedded in the FINVOICE RowIdentifier for each line item, without requiring users to download and parse XML manually.

---

### Gap: AC11 — No enforcement routing configuration UI

**Layer:** Frontend
**AC:** AC11 — Customizable enforcement handling (separate enforcement authorities)
**What is missing:** There is no dedicated screen for configuring enforcement routing — no "enforcement authority" configuration, no "collection process" routing settings, and no per-classification enforcement channel configuration found in the frontend. The publicLawCode and privateLawCode values in CostCenterCompositionConfig are backend-only configuration with no UI to view or update them.
**UI surface needed:** A section on the company settings or cost center configuration page that exposes the Public Law enforcement code and Private Law enforcement code fields, allowing administrators to configure which enforcement authority identifier is sent in the FINVOICE output for each classification type.
