## Gaps

### Gap 1 — AC3: Role-based access not enforced in the UI (multi-provider / role / error AC)

**Layer:** Frontend

**AC:** AC3 — Multi-provider compatibility, role-based access, and error handling

**What is missing from the UI:** The "View Invoice Image (PDF)" button is shown to all users regardless of role. The requirement states that only authorised users may view invoice images. While the backend enforces this via Spring Security (a 403 is returned for unauthorised users), the UI does not proactively hide the button or display a "you do not have permission" message — it only shows a generic error after a failed fetch. A proper implementation would check the current user's roles on page load and either hide the button or show a disabled state with an explanatory tooltip.

**UI surface description:** `InvoiceTransmitPanel` — the "View Invoice Image (PDF)" button at Invoice Detail → Transmission & Integration panel. Does not import or call `getMyRoles()` / `me.js` to gate visibility.

### Gap 2 — AC7: "View Invoice Image (PDF)" button not gated by role in the frontend

**Layer:** Frontend

**AC:** AC7 — Only authorised users can view invoice images

**What is missing from the UI:** The frontend `InvoiceTransmitPanel` does not import or use `getMyRoles()`. Role-based visibility is enforced only at the API layer. A user with an unauthorised role will see the button, click it, and receive a red error message — rather than never seeing the button at all. This represents a UX gap against the requirement.

**UI surface description:** `InvoiceTransmitPanel` — "View Invoice Image (PDF)" button is rendered unconditionally for all authenticated users who can open an invoice detail page. There is no call to `/api/v1/me/roles` or any role check before rendering this control.
