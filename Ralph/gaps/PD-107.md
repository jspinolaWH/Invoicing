## Gaps Found

### Gap 1: Direct debit START order not processed
- **Layer**: Backend + Frontend
- **Criterion**: AC 1 â€” direct debit order integration
- **What exists**: `POST /api/v1/integration/direct-debit-orders` accepts `DirectDebitOrderRequest` with `mandate` and `bankAccount` fields; customer matching works; TERMINATE path does change delivery method to PAPER; START path only logs and returns "Direct debit order received" without further action
- **What is missing**: START case must (a) change `deliveryMethod` to `DIRECT_PAYMENT`, (b) persist mandate and bankAccount (requires a `DirectDebitMandate` entity and DB table or fields on `BillingProfile`), (c) audit the change. Frontend also has no section to display direct debit mandate information on the billing profile page â€” `DIRECT_PAYMENT` is in the delivery method dropdown but selecting it shows no additional fields.
- **Files to change**:
  - `invoicing/src/main/java/com/example/invoicing/service/EInvoiceOrderIngestionService.java` â€” implement START branch in `ingestDirectDebitOrder()` to call changeDeliveryMethod(DIRECT_PAYMENT) and persist mandate
  - `invoicing/src/main/java/com/example/invoicing/entity/customer/` â€” add `DirectDebitMandate` entity or fields on `BillingProfile`
  - `invoicing/src/main/resources/db/migration/` â€” new migration for direct debit storage
  - `invoicing-fe/src/pages/customers/BillingProfilePage.jsx` â€” add direct debit info section when deliveryMethod is DIRECT_PAYMENT

### Gap 2: E-invoice address not created for new customers via integration
- **Layer**: Backend
- **Criterion**: AC 4 â€” e-invoice address added for customer
- **What exists**: `EInvoiceAddressService.updateFromOperator()` (lines 46-50) finds an existing unlocked address and updates it; if no address record exists it falls through to the locked-address warning check and silently does nothing
- **What is missing**: When a START order arrives for a customer with no existing e-invoice address, a new `EInvoiceAddress` record must be created. The `updateFromOperator` method needs a create-if-absent path when neither an unlocked nor locked record exists.
- **Files to change**:
  - `invoicing/src/main/java/com/example/invoicing/service/EInvoiceAddressService.java` â€” in `updateFromOperator()`, when no existing record is found at all, inject `CustomerBillingProfileRepository`, look up the customer, create a new `EInvoiceAddress`, and save it

### Gap 3: Daily import job is a placeholder with no pull mechanism; no run history visible in UI
- **Layer**: Backend + Frontend
- **Criterion**: AC 9 â€” once-per-day delivery from billing system
- **What exists**: `EInvoiceOrderImportJob` is scheduled at `0 0 3 * * *`; `importPendingOrders()` logs "no external source configured â€” awaiting push via POST /api/v1/integration/einvoice-orders" and returns. The E-Invoice Operator Management page displays static text "The daily batch runs automatically at 02:00" and a manual trigger button that shows a one-shot result summary (Started / Terminated / Failed counts for that single run).
- **What is missing**:
  - **Backend**: Either (a) actual pull-based polling or file-drop ingestion logic with a configurable billing-system endpoint URL, or (b) a documented decision that push-only is the agreed delivery model with the placeholder job removed/disabled to avoid confusion.
  - **Frontend**: The E-Invoice Operator Management page has no history of previous automatic or manual batch runs â€” the user cannot see when the last daily batch executed, what it processed, or whether it succeeded. A batch run history table (timestamp, trigger type, Started/Terminated/Failed counts) would make the once-per-day delivery observable without backend log access. The batch schedule time (02:00) is hard-coded in static text; if the schedule changes in the backend the UI will silently show the wrong time.
- **Files to change**:
  - `invoicing/src/main/java/com/example/invoicing/service/EInvoiceOrderImportJob.java` â€” implement pull logic or remove placeholder and document push-only model
  - `invoicing/src/main/resources/application.properties` â€” add integration endpoint config property if pull model is chosen
  - `invoicing-fe/src/pages/integration/OperatorManagementPage.jsx` â€” add a batch run history table that fetches previous run records from a new API endpoint; fetch the scheduled time from the API rather than hard-coding it
