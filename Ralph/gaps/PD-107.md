## Gaps Found

### Gap 1: Direct debit START order not processed
- **Layer**: Backend + Frontend
- **Criterion**: AC 1 — direct debit order integration
- **What exists**: `POST /api/v1/integration/direct-debit-orders` accepts `DirectDebitOrderRequest` with `mandate` and `bankAccount` fields; customer matching works; TERMINATE path does change delivery method to PAPER; START path only logs and returns "Direct debit order received" without further action
- **What is missing**: START case must (a) change `deliveryMethod` to `DIRECT_PAYMENT`, (b) persist mandate and bankAccount (requires a `DirectDebitMandate` entity and DB table or fields on `BillingProfile`), (c) audit the change. Frontend also has no section to display direct debit mandate information on the billing profile page.
- **Files to change**:
  - `invoicing/src/main/java/com/example/invoicing/service/EInvoiceOrderIngestionService.java` — implement START branch in `ingestDirectDebitOrder()`
  - `invoicing/src/main/java/com/example/invoicing/entity/customer/` — add `DirectDebitMandate` entity or fields on `BillingProfile`
  - `invoicing/src/main/resources/db/migration/` — new migration for direct debit storage
  - `invoicing-fe/src/pages/customers/BillingProfilePage.jsx` — add direct debit info section when deliveryMethod is DIRECT_PAYMENT

### Gap 2: E-invoice address not created for new customers via integration
- **Layer**: Backend
- **Criterion**: AC 4 — e-invoice address added for customer
- **What exists**: `EInvoiceAddressService.updateFromOperator()` finds an existing unlocked address and updates it; if no address record exists it falls through to the locked-address warning check and silently does nothing
- **What is missing**: When a START order arrives for a customer with no existing e-invoice address, a new `EInvoiceAddress` record must be created. The `updateFromOperator` method needs a create path when no record exists.
- **Files to change**:
  - `invoicing/src/main/java/com/example/invoicing/service/EInvoiceAddressService.java` — add create-if-absent logic in `updateFromOperator()` (inject `CustomerBillingProfileRepository` already available, create and save new `EInvoiceAddress`)

### Gap 3: Daily import job is a placeholder with no pull mechanism
- **Layer**: Backend
- **Criterion**: AC 9 — once-per-day delivery from billing system
- **What exists**: `EInvoiceOrderImportJob` is scheduled at `0 0 3 * * *`; `importPendingOrders()` logs "no external source configured" and returns
- **What is missing**: Actual pull-based polling or file-drop ingestion logic; a configurable billing-system endpoint URL; HTTP client invocation of the intermediary API (Ropo or similar); parsing of the response and calling `ingestionService.ingestEInvoiceOrder / ingestDirectDebitOrder` per order. If push-only (POST endpoint) is the agreed delivery model, this job body should be documented accordingly and the job should be removed or disabled to avoid confusion.
- **Files to change**:
  - `invoicing/src/main/java/com/example/invoicing/service/EInvoiceOrderImportJob.java` — implement pull logic or remove placeholder and document push-only model
  - `invoicing/src/main/resources/application.properties` — add integration endpoint config property if pull model is chosen
