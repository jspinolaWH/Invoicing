## Acceptance Criteria Coverage

| # | Criterion | Status | Notes |
|---|-----------|--------|-------|
| 1 | The status of invoices created from events must be visible at the event level. | IMPLEMENTED | `BillingEvent.status` field exposed in API; `StatusBadge` used in `BillingEventDetailPage` and `BillingEventsPage` list view |
| 2 | Default status categories: In progress, Sent, Error, Completed (with defined semantics) | IMPLEMENTED | `BillingEventStatus` enum covers IN_PROGRESS, SENT, ERROR, COMPLETED (plus DRAFT, PENDING_TRANSFER); `StatusBadge` maps all to labels |
| 3 | Events → SENT on transfer; SENT events updatable/re-sendable; → COMPLETED on confirmation | IMPLEMENTED | `InvoiceTransmissionService` auto-transitions to SENT/COMPLETED/ERROR; `BilledEventCorrectionService` handles re-invoice flow |
| 4 | Events have process-based statuses that update automatically as workflow progresses | IMPLEMENTED | `BillingEventStatusService.ALLOWED_TRANSITIONS` enforces state machine; `InvoiceTransmissionService` triggers auto-transitions |
| 5 | Event statuses visible in both summary view and customer-specific information | IMPLEMENTED | `BillingEventsPage` provides summary + status counts strip; `BillingProfilePage` now shows per-customer event table with `StatusBadge` and per-status counts |
| 6 | Events editable until marked completed and invoice successfully transferred | IMPLEMENTED | `assertMutable()` on backend; `canEdit` guard on frontend blocks edits for SENT/COMPLETED |
| 7 | Incorrect event data correctable after invoicing run if not yet received by external system | IMPLEMENTED | `BilledEventCorrectionService.correct()` issues credit note and resets events to IN_PROGRESS for re-processing |
