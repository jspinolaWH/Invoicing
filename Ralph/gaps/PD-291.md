## Gaps Found

### Gap 1 — AC2: No per-billing-period event grouping view in simulation results
**AC:** AC2 — Events grouped by billing period and validated before invoice generation
**Layer:** Frontend
**What is missing:** The Simulation Results page does not display events grouped explicitly by their assigned billing period (e.g., "Q1 events for Customer X"). The backend correctly groups events by period before invoicing, but the simulation results UI presents only aggregate totals and category breakdowns. There is no per-cycle breakdown showing which billing period each event belongs to, making it impossible for a user to verify that grouping occurred correctly without examining the raw invoice output after a real run.

---

### Gap 2 — AC5: No per-cycle event grouping report in the UI
**AC:** AC5 — Automatic event grouping sorted and billed according to their assigned schedules
**Layer:** Frontend
**What is missing:** No dedicated UI screen or panel shows which events were grouped into which billing cycle's invoice. The Simulation Results page and Invoice Run Detail page present only aggregate invoice counts and category summaries. A user cannot drill down to see "these 7 events from Customer X were batched into the Q2 QUARTERLY cycle invoice" — the per-cycle grouping is backend-driven and entirely invisible to users until they compare individual invoice line items manually.

---

### Gap 3 — AC8: No UI surface for external financial/accounting system integration
**AC:** AC8 — Seamless integration with external financial and accounting systems
**Layer:** Frontend
**What is missing:** There is no user-facing screen, status indicator, or configuration panel specifically for billing-cycle-to-accounting-system integration. After a real invoice run completes, users have no way to confirm from the UI that cycle-based invoices were successfully transmitted to the external financial system, or to see any integration errors. The accounting and ERP integration configuration is partially visible in Master Data (e-invoice operators), but there is no run-level integration status tied to billing cycles.

---

### Gap 4 — AC12: No per-cycle attribution on simulation or run results pages
**AC:** AC12 — ERP consolidates summer cottage events into single annual invoice in December
**Layer:** Frontend
**What is missing:** The simulation results and run detail pages do not display per-cycle attribution — users cannot see which billing cycle (e.g., "ANNUAL cycle for property SUMMER_COTTAGE_REF") generated a given invoice. The consolidation logic runs correctly in the backend, but the UI confirms it only via the total invoice count. There is no label, badge, or breakdown that identifies an invoice as "generated from the ANNUAL cycle for property X," forcing users to infer consolidation occurred from the invoice count alone.
