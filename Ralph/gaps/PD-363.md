## Frontend UI Gaps

---

### Gap 1 — AC6: "Update Service Responsibility" does not pre-populate customer or link back to trigger

**Layer:** Frontend

**AC:** AC6 — Users can update service responsibility or document justification for retaining current classification

**What is missing:** The "Update Service Responsibility" button on the Trigger Detail page navigates the user to the general Responsibility Change page (Tools → Resp. Change), but it does not pre-populate the customer number in the "From Customer Number" field or pass any reference back to the originating trigger. The user must manually type the customer number again and there is no mechanism to record on the trigger that a responsibility change was subsequently performed. The handoff between the trigger workflow and the classification update is entirely manual with no data continuity.

### Gap 2 — AC7: No unified audit trail linking trigger review decisions to responsibility change records

**Layer:** Frontend

**AC:** AC7 — All trigger events, review decisions, and responsibility changes are recorded in a full audit trail

**What is missing:** The Trigger Detail page shows an audit trail for the trigger itself (creation timestamp, review timestamp, decision text) and the Responsibility Change page has its own separate change history log. However, there is no UI link between the two: when a trigger review leads to a responsibility change, the trigger detail does not show "Responsibility changed on [date] by [user]" and the change log does not reference the trigger ID that prompted the change. Auditors see two disconnected audit views with no traceability between them. A combined or cross-linked audit view is needed to confirm the full review-to-action lifecycle.
