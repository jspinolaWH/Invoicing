## Gaps Found

### Gap 1: Restricted billing address and billing group fields not visually disabled pre-emptively
- **Layer**: Frontend
- **Criterion**: AC 2 / AC 5 — dynamic field classification; billing address locked during active run
- **What exists**: The backend correctly enforces the lock and returns HTTP 423 when a save is attempted on a locked customer. The `CustomerLockedBadge` component renders an amber badge on the Billing Profile page with a tooltip explaining which fields are restricted.
- **What is missing**: The billing address input fields (Street Address, Postal Code, City, Country Code) are not visually disabled or greyed out when the customer is locked. The user must attempt to save before receiving the rejection. The requirement specifies that fields should be dynamically classified as editable or restricted — the UI should pre-emptively disable or mark restricted fields when the lock badge is present, eliminating the need for the user to attempt a rejected save.
- **Files to change**:
  - `invoicing-fe/src/pages/customers/BillingProfilePage.jsx` — when the `isLocked` state is true, set `disabled` on the billing address fields (streetAddress, postalCode, city, countryCode) and apply a visual "locked" style (e.g. greyed background, lock icon) alongside each field

### Gap 2: Bundling Rules page shows no lock indicator or field disabling
- **Layer**: Frontend
- **Criterion**: AC 6 — billing groups cannot be modified during active run
- **What exists**: The backend correctly rejects bundling rule saves for locked customers via `PropertyGroupService.assertNotLocked()`. The Billing Profile page displays the `CustomerLockedBadge` when the customer is locked.
- **What is missing**: The Bundling Rules page (`BundlingRulesPage.jsx`) does not import or render the `CustomerLockedBadge` component. There is no lock indicator, no disabled state on the bundling rule inputs, and no inline warning on this page — the user only discovers the lock after a failed save attempt. The lock badge and pre-emptive field disabling should be replicated here as they are on the Billing Profile page.
- **Files to change**:
  - `invoicing-fe/src/pages/customers/BundlingRulesPage.jsx` — import and render `CustomerLockedBadge`; fetch lock status using the existing `locks.js` API; disable bundling rule inputs and the Save button when `isLocked` is true

### Gap 3: No app-wide active run notification banner
- **Layer**: Frontend
- **Criterion**: AC 9 — real-time alert when a billing run is active
- **What exists**: The `CustomerLockedBadge` renders an amber badge on the individual customer's Billing Profile page when that specific customer is locked by an active run. The Invoice Run Detail page shows the run status in real time.
- **What is missing**: There is no application-wide banner or persistent notification that informs any logged-in user that a billing run is currently in progress. Users browsing billing events, invoices, or other sections of the app have no indication that a run is active. The requirement calls for the system to alert users when a billing run is active — a global run-status indicator (e.g. a persistent top-bar banner or notification chip) would satisfy this for all pages, not just the billing profile of locked customers.
- **Files to change**:
  - `invoicing-fe/src/components/Layout.jsx` (or equivalent top-level layout component) — add a global active-run banner that polls `GET /api/v1/runs/active` (or equivalent) and renders an amber notification bar when a run is in RUNNING status, dismissible per session
