## Frontend Gaps (added from UI synthesis)

---

### Gap: AC3 — No company-wide default configuration UI

**AC:** Default gross/net settings ensure consistency and reduce manual adjustments
**Layer:** Frontend
**UI Surface:** Billing Profile page (Customers & Config → Billing Profiles)
**What is missing:** The backend defaults to NET and the new-profile form initialises with NET pre-selected, but there is no company-wide or system-level screen where an administrator can view or change the global default invoicing mode. Users cannot see or modify a global default — only the per-customer setting. The default is hardcoded in the form with no administrative override surface.

---

### Gap: AC4 — No visible role gate on the Invoicing Mode override field

**AC:** Authorized users can override the default setting at the customer level
**Layer:** Frontend
**UI Surface:** Billing Profile page → Invoicing Mode dropdown
**What is missing:** The Invoicing Mode dropdown is editable by any logged-in user who can view a customer's Billing Profile. No visible role-based restriction, disabled state, or tooltip is shown to communicate that only authorized users may change this field. The backend enforces access control via `@PreAuthorize("hasRole('INVOICING')")`, but the frontend provides no UI-level indication of this restriction.

---

### Gap: AC8 — No UI-level access control on gross/net setting

**AC:** Only authorized personnel can modify the gross/net setting
**Layer:** Frontend
**UI Surface:** Billing Profile page → Invoicing Mode dropdown
**What is missing:** Any authenticated user who navigates to a customer's Billing Profile can see and interact with the Invoicing Mode dropdown. There is no frontend role check, read-only state, lock icon, or informational tooltip indicating that modification requires the INVOICING role. The UI does not reflect the backend access control enforced by the `@PreAuthorize` annotation on the PUT endpoint.
