## Frontend UI Gaps

---

### Gap 1 — AC2: Unclear distinction between Change Effective Date and event date filters

**Layer:** Frontend

**AC:** AC2 — Users can define a start date and the system identifies and updates all related uninvoiced events

**What is missing:** The Responsibility Change form presents both a "Change Effective Date" field and "Event Date From / Event Date To" filters, but provides no explanation of how these two date concepts differ. The effective date appears to be stored as audit metadata only, while the event date filters perform the actual scoping. A non-technical user has no way to understand this from the UI alone — they may assume the effective date drives event selection when it does not. A tooltip, inline help text, or label clarification is needed to distinguish "metadata date stored with the change run" from "date range that filters which events are included."

**UI Surface:** `invoicing-fe/src/pages/retroactive/ResponsibilityChangePage.jsx`

---

### Gap 2 — AC8: Responsibility Change History report cannot be exported or filtered by date range

**Layer:** Frontend

**AC:** AC8 — Users can generate reports based on original or updated responsibility for internal and external audits

**What is missing:** The Responsibility Change History table (loaded via "Load Report") shows audit data per change run, but the UI provides no way to export the report (e.g. CSV or PDF download) and no date-range filter on the history itself. Auditors who need a structured document covering a specific period across all customers cannot produce one from the UI. The API supports filtering by customer number only; date-scoped or cross-customer export is not surfaced at all.

**UI Surface:** `invoicing-fe/src/pages/retroactive/ResponsibilityChangePage.jsx`

---

### Gap 3 — AC15: Seasonal same-customer scenario has confusing UX and no period concept

**Layer:** Frontend

**AC:** AC15 — Seasonal scenario: update seasonal service retroactively for a specific summer period

**What is missing:** The UI requires both "From Customer Number" and "To Customer Number" fields even when the responsibility holder remains the same and only the classification changes. It is unclear whether identical customer numbers in both fields are accepted or rejected by the form. Additionally, there is no "seasonal period" concept, template, or schedule helper — users must manually enter date ranges with no guidance. A same-customer classification change (only the serviceResponsibility changes, not the customer account) may not map intuitively onto the From/To form layout, risking user error.

**UI Surface:** `invoicing-fe/src/pages/retroactive/ResponsibilityChangePage.jsx`
