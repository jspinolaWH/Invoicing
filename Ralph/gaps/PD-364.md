| # | Criterion | Status | Notes |
|---|-----------|--------|-------|
| 1 | Retroactive changes applied to service records, reports, and invoicing tools so responsibility classifications remain accurate | IMPLEMENTED | BillingEvent.serviceResponsibility updated in-place for in-progress events; correction copies created for billed events; audit log and change log capture everything |
| 2 | Users can define a start date for when the change takes effect; system updates all related uninvoiced events | IMPLEMENTED | `changeEffectiveDate` field in `ResponsibilityChangeRequest`; date-range filtering in `resolveEvents()`; FE date picker present; INVOICING role required |
| 3 | Change automatically reflected in linked processes: service event logs, eligibility/pricing, compliance reporting | IMPLEMENTED | Audit log records old/new serviceResponsibility per event; `ServiceResponsibilityChangeEventHandler` @TransactionalEventListener logs structured summary; serviceResponsibility field used for municipal/market categorisation |

| 6 | System logs all retroactive changes with user, timestamps, and events affected | IMPLEMENTED | `ServiceResponsibilityChangeLog` + per-event `BillingEventAuditLog`; appliedBy, appliedAt, affectedCount, reason all captured |

| 9 | Waste companies can correct administrative data while ensuring accuracy and compliance | IMPLEMENTED | Full preview→confirm→apply workflow; audit trail; correction copies for billed events; role guard |
| 10 | Functionality supports waste company administration, compliance officers, and service managers | IMPLEMENTED | Date-range, customer, product, and specific-event-ID targeting all supported |
| 11 | Scenario A: municipality transfers commercial responsibility retroactively to regional authority | IMPLEMENTED | Customer number + serviceResponsibility change from effective date; uninvoiced events reclassified; billed events get correction copies |
| 12 | Scenario B: TSV responsibility corrected retroactively for a business customer | IMPLEMENTED | fromCustomerNumber + newServiceResponsibility + eventDateFrom corrects pending events; previously invoiced events unaffected |
| 13 | Scenario C: seasonal service change reflected in service log and compliance reports | IMPLEMENTED | eventDateFrom/eventDateTo scopes the change to the agreed summer period; audit log and change log record the correction |
