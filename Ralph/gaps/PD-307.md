## Gaps Found

### Gap 1: InvoiceRunResponse missing templateId and numberSeriesId fields

- **Layer**: Backend
- **Criterion**: AC5 — template selection during an invoicing run
- **What exists**: `InvoiceRun.templateId` (line 101) and `InvoiceRun.numberSeriesId` (line 98) are stored on the entity. `InvoiceRunRequest.templateId` accepts the value on creation. `RunFilterForm.jsx` sends `templateId` when creating a run. `InvoiceRunService` correctly stores both values. `InvoiceGenerationService.resolveTemplateCode()` correctly uses the run-level `templateId` as the highest priority in the resolution hierarchy (run → contract → property → customer → default).
- **What is missing**: `InvoiceRunResponse` (line 11) does not expose `templateId` or `numberSeriesId` fields. The `from()` mapper (line 30) does not map these fields. The frontend cannot read back which template or number series was applied to a run after creation.
- **Files to change**:
  - `invoicing/src/main/java/com/example/invoicing/entity/invoicerun/dto/InvoiceRunResponse.java` — add `private Long templateId;` and `private Long numberSeriesId;` fields, update `from()` mapper to include both
