## Frontend Gaps (added from UI synthesis)

---

### Gap: AC8 — No UI surface for FINVOICE 3.0 compliance visibility

**AC:** Invoice data file must comply with the FINVOICE 3.0 standard
**Layer:** Frontend
**UI Surface:** Invoice Detail page / Generate Invoice page
**What is missing:** There is no user-facing screen for FINVOICE 3.0 compliance configuration or inspection. Users cannot view the generated FINVOICE XML, see which fields are mapped to FINVOICE 3.0 elements, or verify that the output file is compliant before transmission. The system produces the compliant file automatically in the backend, but the UI provides no transparency — no preview, no download of the raw FINVOICE XML from the invoice detail page, and no compliance indicator. Users must infer correctness from SENT/ERROR transmission statuses alone.

---

### Gap: AC11 — No dedicated per-company mandatory fields configuration UI

**AC:** Waste management companies define information guiding invoicing data generation
**Layer:** Frontend
**UI Surface:** Customers & Config → Validation Rules
**What is missing:** The Validation Rules page provides a general rule engine mechanism for adding custom validation checks, but does not expose a dedicated "additional mandatory fields per company" workflow. There is no structured UI where an administrator can see a clear per-company mandatory field matrix, add new company-specific required fields from a field catalogue, or understand which fields are mandatory for their company versus the baseline. The requirement implies a targeted per-company field definition screen that is not present — only the general validation rule engine is available.

---

### Gap: AC12 — No UI for FINVOICE baseline field configuration or deviation management

**AC:** Basic FINVOICE data defined as common for all companies; deviations handled via change request
**Layer:** Frontend
**UI Surface:** No screen exists
**What is missing:** There is no user-facing screen for viewing the FINVOICE baseline field set that is common across all companies, nor for initiating or tracking a change request to request a company-specific deviation. The requirement implies users or administrators should be able to see which FINVOICE fields are standard and which are company-specific, and have a workflow for requesting deviations. This is currently an operational/contractual process handled entirely outside the application with no corresponding UI representation.
