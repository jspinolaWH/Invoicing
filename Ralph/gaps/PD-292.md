## Gaps Found

### Gap 1 — AC5: No conflict-prevention visibility in the UI
**AC:** AC5 — Invoice grouping; prevent conflicting groupings (one-off not delayed by recurring)
**Layer:** Frontend
**What is missing:** The Bundling Rules page shows grouping configuration, but there is no UI surface that confirms or makes visible the conflict-prevention logic — specifically, that IMMEDIATE billing-type events are shielded from cycle delays. There is no screen, badge, banner, or indicator showing users that a given event will be processed immediately rather than held for a cycle. The backend enforces this correctly, but the user has no way to observe or verify the enforcement from the UI without reading documentation.

---

### Gap 2 — AC7: No UI surface for external financial tool integration
**AC:** AC7 — Integration with external financial management tools
**Layer:** Frontend
**What is missing:** There is no dedicated UI screen, status panel, or configuration page for the integration with external financial and ERP systems. Users cannot see integration status, recent sync events, or configuration options for external dispatch. The backend integration layer handles this automatically, but there is zero frontend visibility into whether invoices were successfully dispatched to external systems or whether any integration errors occurred.

---

### Gap 3 — AC10: No per-event runtime enforcement visibility
**AC:** AC10 — System generates immediate invoice for septic tank; semi-annual for waste
**Layer:** Frontend
**What is missing:** After rules are configured, there is no UI surface where a user can observe that a specific billing event was invoiced immediately due to a restriction (as opposed to being held for a cycle). The result of the billing-type enforcement is only visible in the aggregate invoice output after a run completes. There is no event-level indicator, timeline view, or run-detail breakdown showing which events were processed immediately vs. cycle-batched, making it impossible for a user to confirm the rules are working correctly without manually cross-referencing invoice runs and event timestamps.
