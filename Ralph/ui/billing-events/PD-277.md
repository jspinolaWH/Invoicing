## Where to Find This Feature

This feature lives under the **Operations** tab. The main entry point is the **Billing Events** list, from which users click **Edit** (directly on a row) or navigate into the event detail page and click **Edit** there. Corrected events awaiting approval show a "FOR_CORRECTION" status and can be reviewed through both the event detail and the **Review Queue** sidebar item.

## User Flows by Acceptance Criterion

---

### AC1 — Authorized users can edit event data; changes are recorded and tracked

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. You will see a table of billing events showing columns for Date, Customer #, Product, Waste Fee, Transport Fee, Eco Fee, Quantity, Classification, Status, Validation, and Origin.
4. Find an event with status **DRAFT**, **IN PROGRESS**, **FOR CORRECTION**, or **ERROR** — only these statuses allow editing.
5. Click the **Edit** button on that row (visible directly in the Actions column for editable events).
6. You land on the **Edit Billing Event** page. The page header states "All changes are logged with a mandatory reason."
7. Make any field changes, then scroll to the **Reason for Edit** section at the bottom and type a reason in the required text box.
8. Click **Save Changes**.
9. You are redirected back to the event detail page.

**Confirms requirement:** The edit form explicitly enforces a mandatory reason field and the page header states all changes are logged, confirming that corrections are recorded and tracked.

**UI Status:** ✅ VISIBLE

---

### AC2 — Editable fields for events prior to invoicing (header criterion)

1. Follow steps 1–5 from AC1 to reach the Edit Billing Event page.
2. The form is organized into clearly labeled sections: **Event Details**, **Pricing**, **Quantity & Weight**, **Customer & Location**, **Comments**, **Waste Classification**, **Additional Details**, and **Reason for Edit**.
3. If the event status is NOT **IN_PROGRESS** or **ERROR**, a red warning banner appears: "This event is [status] and cannot be edited." All fields remain visible but no Save button is shown.

**Confirms requirement:** The form exposes all the editable fields described in the requirement for pre-invoicing events, grouped by category.

**UI Status:** ✅ VISIBLE

---

### AC3 — Product quantities (correcting an incorrect number of containers)

1. Follow steps 1–5 from AC1 to reach the Edit Billing Event page.
2. Scroll to the **Quantity & Weight** section.
3. Edit the **Quantity** field (decimal, e.g. change 50 to 5).
4. Optionally also update the **Weight** field.
5. Scroll down, enter a reason in **Reason for Edit**, and click **Save Changes**.

**Confirms requirement:** The Quantity field is directly editable on the form, matching the example of correcting an incorrect container count.

**UI Status:** ✅ VISIBLE

---

### AC4 — Event type (changing a misclassified event)

1. Follow steps 1–5 from AC1 to reach the Edit Billing Event page.
2. In the **Event Details** section, locate the **Product** dropdown.
3. Select a different product from the list (the product represents the service/event type, e.g. switching from "container emptied" to a different service code).
4. Enter a reason and click **Save Changes**.

**Confirms requirement:** The Product dropdown allows changing the event classification/type, covering the misclassified event correction scenario.

**UI Status:** ✅ VISIBLE

---

### AC5 — Contractor fees and adjustments

1. Follow steps 1–5 from AC1 to reach the Edit Billing Event page.
2. Scroll to the **Additional Details** section.
3. Edit the **Contractor** field (the name or identifier of the contractor).
4. On the event detail page (after saving), if a contractor is set, a **Contractor Payment** section appears at the bottom of the Details tab.
5. Click **Record Payment Decision** to set the payment status to **Paid** or **Not Required** and optionally add notes.

**Confirms requirement:** The contractor field is editable, and a dedicated Contractor Payment section on the detail page allows recording payment decisions and adjustments.

**UI Status:** ✅ VISIBLE

---

### AC6 — Customer price (waste fee, transport fee, eco fee)

1. Follow steps 1–5 from AC1 to reach the Edit Billing Event page.
2. Scroll to the **Pricing** section.
3. Edit **Waste Fee**, **Transport Fee**, and/or **Eco Fee** fields (decimal inputs).
4. Note: users with the **INVOICING_PRICING** role (pricing-only access) can edit these three fields even if they cannot edit other fields.
5. Enter a reason and click **Save Changes**.
6. On the event detail page, if prices were overridden, a **Price Override** panel appears showing the product default price alongside the new price (e.g. "Product default: €10.00 → €8.50").

**Confirms requirement:** All three price component fields are editable, and price overrides are highlighted on the detail page for transparency.

**UI Status:** ✅ VISIBLE

---

### AC7 — Real-time correction workflow: access list of pre-invoicing events

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. You see a full list of events. Use the **Status** dropdown filter to select **IN PROGRESS** (or other pre-invoicing statuses like **DRAFT**) and click **Apply**.
4. The list narrows to show only pre-invoicing events available for correction.
5. The **Edit** button appears directly on each editable row, enabling direct access to corrections.

**Confirms requirement:** The Billing Events list with status filtering provides a real-time view of pre-invoicing events ready for correction.

**UI Status:** ✅ VISIBLE

---

### AC8 — System automatically updates financial calculations after correction

1. Follow the steps from AC3 or AC6 to edit quantities or prices.
2. After saving, navigate back to the event detail page.
3. Scroll to the **VAT Details** panel (shown automatically when VAT data is present). The Net Amount, VAT Amount, and Gross Amount fields reflect the updated prices.
4. The "Total with 0% VAT" and "Total with 24% VAT" calculated values also update to reflect the changed prices or quantities.

**Confirms requirement:** The event detail page automatically shows recalculated financial amounts based on the corrected data, confirming the system recalculates before invoicing.

**UI Status:** ✅ VISIBLE

---

### AC9 — Corrected events flagged (FOR_CORRECTION) and blocked from invoicing until reviewed

1. When a user edits an event that is in **IN_PROGRESS** status and saves it, the system automatically sets the event status to **FOR_CORRECTION**.
2. On the event detail page, a purple banner appears: "Awaiting correction review — This event was manually edited and is blocked from invoicing until an INVOICING user approves the correction."
3. The **Approve Correction** button appears at the top of the event detail page (visible only to users with the INVOICING role).
4. On the Billing Events list, the stat strip at the top shows a count for **For Correction** events, allowing supervisors to monitor the correction queue.
5. Click **Approve Correction** to re-release the event into the invoicing flow.

**Confirms requirement:** The FOR_CORRECTION status badge and the blocking banner confirm that corrected events are flagged and cannot proceed to invoicing until approved.

**UI Status:** ✅ VISIBLE

---

### AC10 — Audit log: user, timestamp, and reason for each change

1. Follow steps 1–4 from AC1 to open a billing event detail page.
2. Click the **Audit Trail** tab (visible in the tab bar below the event header: Details | Audit Trail | Status | Attachments).
3. A table appears showing every recorded change with columns: **Timestamp**, **User**, **Field**, **Previous Value**, **New Value**, and **Reason**.
4. Use the filter bar to narrow results by user name, field changed, or date range.

**Confirms requirement:** The Audit Trail tab shows a complete record of who changed what and when, including the mandatory reason entered by the editor.

**UI Status:** ✅ VISIBLE

---

### AC11 — Before-and-after records for transparency

1. Navigate to the Audit Trail tab as described in AC10.
2. Each row in the audit table shows both **Previous Value** and **New Value** side by side for every changed field.
3. For price overrides specifically, the event detail page's **Price Override** panel also displays the original product default alongside the overridden price.

**Confirms requirement:** Before-and-after values are shown explicitly in the Audit Trail table, providing the transparency and reference data required for customer inquiries or audits.

**UI Status:** ✅ VISIBLE

---

### AC12 — Only authorized personnel have access to manual corrections

1. On the Edit Billing Event page, the system calls the current user's roles via the API.
2. If the event status is not **IN_PROGRESS** or **ERROR**, the page displays a red error message: "This event is [status] and cannot be edited. Use the credit-and-re-invoice flow for corrections." — and the **Save Changes** button is hidden.
3. The **Edit** button on the Billing Events list is only rendered for rows with editable statuses (DRAFT, IN_PROGRESS, FOR_CORRECTION, ERROR) — other rows show only a **View** button.
4. On the detail page, the **Edit** button is disabled and shows a tooltip explaining the event cannot be edited when the status is non-editable.

**Confirms requirement:** Edit access is controlled by event status and user role; unauthorized users or non-editable events do not expose an active edit path.

**UI Status:** ✅ VISIBLE

---

### AC13 — Role-based permission levels (pricing-only vs full edit access)

1. Navigate to the Edit Billing Event page for an IN_PROGRESS event.
2. If the logged-in user has the **INVOICING_PRICING** role (but not the full **INVOICING** role), a yellow information banner appears: "Your account has Pricing-only edit access. Only the Pricing fields (Waste Fee, Transport Fee, Eco Fee) can be modified."
3. In this mode, all non-pricing fields (Date, Product, Quantity, Weight, Customer, Location, etc.) are disabled (greyed out) and cannot be changed.
4. Only the **Waste Fee**, **Transport Fee**, and **Eco Fee** fields remain active.
5. Users with the full **INVOICING** role see all fields active with no restriction banner.

**Confirms requirement:** The pricing-only permission banner and the selective field disabling demonstrate role-based access control within the edit form.

**UI Status:** ✅ VISIBLE
