## Where to Find This Feature

This feature lives primarily under the **Operations** tab. The billing event schema is visible both when creating a new event (via the **+ New Event** button on the Billing Events list) and when viewing any existing event's detail page. Almost all schema fields are shown in the **Details** tab of the event detail page.

## User Flows by Acceptance Criterion

---

### AC1 — View the full billing event schema in the UI

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. You will see a table listing all billing events with columns for Date, Customer, Product, Origin, and Status.
4. Click the **View** button on any event row to open its detail page.
5. You will land on the **Details** tab automatically. The page shows a grid of labeled fields covering all schema elements:
   - **Date** — the event date
   - **Customer #** — the customer number
   - **Product** — product name or code (maps to Product/Service and Product name/Service name)
   - **Origin** — shown as a badge
   - **Waste Fee**, **Transport Fee**, **Eco Fee** — the three price formation components
   - **Quantity** — in the configured unit (pcs, m³, etc.)
   - **Weight** — in tonnes or kg
   - **VAT 0%** and **VAT 24%** — individual VAT rate fields
   - **Registration Number** — vehicle registration
   - **Accounting Account** — linked account with code and name
   - **Cost Center** — composite code
   - **Municipality**, **Location**, **Vehicle**, **Driver** — location-related fields
   - **Contractor** — the contractor name
   - **Direction** — INBOUND or OUTBOUND
   - **Shared Collection Group %** — the property group percentage
   - **Service Responsibility** — shown in the field grid
   - **Non-billable** — Yes/No flag for events not intended to be invoiced
6. Scroll down to see the **VAT Details** panel which shows:
   - VAT Rate Code, Effective Rate, Reverse Charge flag
   - Net Amount, VAT Amount, Gross Amount
   - **Total with 0% VAT** and **Total with 24% VAT** — computed totals
7. Further below, **Invoice Comments** (maps to Event comments / free-text comment field) and **Internal Notes** appear if they have been set.
8. The event also shows a unique system-assigned **Event #** in the page heading (maps to Event identifier).

**Confirms requirement:** The detail page grid displays every schema field listed in the requirement, with values shown for fields that have been populated and "—" for empty optional fields.

**UI Status:** ✅ VISIBLE

---

### AC2 — Events store data for invoice generation; events can be marked non-billable

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. Click **View** on any event row.
4. On the **Details** tab, scroll down to the **VAT Details** panel. You will see the resolved VAT rate code and effective percentage tied to the event's date, confirming the system resolves and stores VAT at time-of-event.
5. To see a non-billable event, look for the **Non-billable** field in the main detail grid. If the field shows **Yes**, the event will not be included in invoice generation.
6. When creating a new event via **+ New Event** on the Billing Events page, the system automatically resolves and stores accounting account, cost center, legal classification, and VAT rates based on the selected product and date — confirmed by the "Auto-Resolved Values" preview block that appears on the create form once a product and date are both entered.

**Confirms requirement:** The VAT Details panel proves the system stores event-date-bound VAT rates; the Non-billable field confirms events can be flagged to exclude them from invoicing.

**UI Status:** ✅ VISIBLE

---

### AC3 — Only relevant fields are generated for each event type

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. Click **+ New Event** to open the Create Billing Event form.
4. Notice that most fields beyond Date, Customer, Product, Waste Fee, Transport Fee, Eco Fee, Quantity, and Weight are marked **optional**. Fields such as Municipality, Location ID, Vehicle ID, Driver ID, Comments, Contractor, Direction, Property Group, Waste Type, and Receiving Site are optional and need not be provided.
5. After creating or viewing an event, open any event's detail page. Fields that were not populated display "—", confirming the system stores only what is relevant and does not require every field for every event.

**Confirms requirement:** The form clearly distinguishes required from optional fields, and the detail page gracefully renders "—" for unpopulated optional fields, confirming that only relevant data is stored per event.

**UI Status:** ✅ VISIBLE

### AC4 — Event data included in FINVOICE material per FINVOICE specification

There is no user-facing screen where the FINVOICE output can be inspected or configured directly. The FINVOICE formatting happens automatically in the backend when an event transitions to the transmission stage. Users can observe that events are transmitted (the status changes to SENT) but there is no UI that exposes the raw FINVOICE XML or lets users see which fields map to which FINVOICE elements.

**Confirms requirement:** Cannot be confirmed visually — this is a backend serialisation concern.

**UI Status:** ❌ NOT VISIBLE

*The FINVOICE generation happens entirely in backend invoice/transmission logic. There is intentionally no UI for this; correctness is verified through backend integration tests and transmission outcomes (SENT/ERROR statuses visible on the event detail Status tab).*

### AC5 — Same invoice can include multiple VAT rates and cost centres

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. Click **View** on an event. On the **Details** tab you can see the **VAT Details** panel showing the resolved VAT rate for that individual event, and the **Cost Center** field.
4. The system supports multiple VAT rates and cost centres across the events that are grouped onto a single invoice, but this is visible at invoice level rather than per-event. To see this, click **Invoices** in the top navigation bar, then **Invoice List** in the sidebar, and open an invoice to see line items grouped by different VAT rates.

**Confirms requirement:** Individual event detail shows the per-event resolved VAT and cost centre. Multiple rates on a single invoice is visible from the invoice level but not surfaced within a single billing event's own detail page.

**UI Status:** ⚠️ PARTIAL

*The per-event VAT and cost centre are visible. The multi-rate, multi-cost-centre view at invoice level exists on the invoice detail page, but the billing event detail page itself does not show how it combines with sibling events. A tester can only confirm multiple rates are supported by inspecting an invoice that has been generated from events with different rate configurations.*

---

### AC6 — Event data generated via weighbridge or cashier/POS integration

Events originating from a weighbridge or POS system will appear in the Billing Events list with an **Origin** badge set to the relevant source (e.g. WEIGHBRIDGE). To see this:

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. Look for events in the list that show an origin badge other than MANUAL. If a weighbridge or POS integration has sent events, they will appear here.
4. Click **View** on such an event. The **Origin** field on the Details tab will confirm where the event came from.

However, there is no UI screen where a user can trigger a weighbridge import, configure POS event ingestion, or simulate an integration-sourced event creation. The integration endpoints are backend-only.

**Confirms requirement:** Integration-sourced events appear in the list with their origin labelled, but the creation/ingestion side of the integration has no user-facing screen.

**UI Status:** ❌ NOT VISIBLE

*Weighbridge and POS integrations push events to the backend via API. There is a WeighbridgeConfigPage accessible under the Tools / integration routes (not yet in the main sidebar) for configuration, but no UI surface for the actual event ingestion process. Integration-generated events are visible once they arrive, but the generation mechanism itself is backend-only.*

## Backend-Only Aspects

- **AC4 — FINVOICE formatting:** The mapping of billing event fields to the FINVOICE XML specification is handled entirely in the backend serialisation layer. There is no UI that exposes FINVOICE output or lets users inspect field mappings. This is intentional — FINVOICE is a machine-readable format for transmission to the operator network, not a user-facing document.

- **AC6 — Weighbridge / POS event creation:** The actual ingestion of events from weighbridge terminals or cashier/POS systems happens via backend integration APIs. There is no UI for triggering or simulating these flows. Once events arrive they are visible in the Billing Events list like any other event.
