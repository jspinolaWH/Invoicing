## Where to Find This Feature

This feature lives primarily under the **Master Data** tab. Accounting accounts are managed under the **Accounting** sidebar item, cost centers under **Cost Centers**, product-to-account mappings under **Allocation Rules**, and per-product defaults under **Products**. Account and cost center data also appears on invoice line items, visible from the **Invoices** tab.

## User Flows by Acceptance Criterion

---

### AC1 — System assigns products to correct accounts based on implementation rules

1. Click **Master Data** in the top navigation bar.
2. In the left sidebar, click **Allocation Rules**.
3. You will see a table of all allocation rules, each row showing: Product, Region, Municipality, Price Component, Account, Specificity, and whether the rule is Active.
4. The system automatically picks the most specific matching rule when generating invoices — no user action is needed at billing time.

**Confirms requirement:** The allocation rules table shows how the system maps each product (optionally by region, municipality, and price component) to a specific accounting account, confirming rule-based routing is in place.

**UI Status:** ✅ VISIBLE

---

### AC2 — Cost center info consists of multiple components (product, reception, responsibility)

1. Click **Master Data** in the top navigation bar.
2. In the left sidebar, click **Cost Centers**.
3. You will see a table with columns: Composite Code, Product (segment), Reception (segment), Responsibility (segment), and Description.
4. Click **+ Add Cost Center** to open the creation form.
5. You will see three required input fields: **Product Segment**, **Reception Segment**, and **Responsibility Segment**, plus an optional **Description** field.
6. As you type into each segment field, a live **Composite code preview** appears at the top of the form, showing the three segments joined with hyphens (e.g., `WASTE-HELSINKI-01-MUNICIPAL`).

**Confirms requirement:** The form explicitly models the three-component cost center structure (product, reception point, and responsibility/contract) that the requirement describes.

**UI Status:** ✅ VISIBLE

---

### AC3 — Structure and content of cost centers defined during implementation; changes via WasteHero support

1. Click **Master Data** in the top navigation bar.
2. In the left sidebar, click **Cost Centers**.
3. Main users can create, edit, and delete cost center definitions using the **+ Add Cost Center**, **Edit**, and **Delete** buttons on each row.

**Confirms requirement:** The cost center management UI is present and accessible to main users. However, there is no explicit UI message or workflow indicating "contact WasteHero support for structural format changes" — this guidance exists only in documentation/implementation processes, not in the UI itself.

**UI Status:** ⚠️ PARTIAL

**What is missing:** The UI allows direct editing of all three segment fields, which means main users can structurally change the format themselves. The AC implies that format changes require support assistance, but there is no UI-level restriction or "contact support" prompt to enforce this. The self-service capability may be intentional or may need a restriction/note added.

### AC4 — Accounts for different products defined in product price lists (VAT, transport, waste treatment)

1. Click **Master Data** in the top navigation bar.
2. In the left sidebar, click **Allocation Rules**.
3. You will see the full list of rules mapping products to accounts, optionally by price component (WASTE_FEE, TRANSPORT_FEE, ECO_FEE, SURCHARGE, ADJUSTMENT), region, and municipality.
4. Click **+ Add Rule** to open the rule creation modal.
5. You will see a **Product** dropdown, a **Price Component** dropdown (which lets you target a specific pricing component like Transport Fee or Waste Fee), an **Accounting Account** search field, and optional region/municipality fields.
6. You can also set a **Default Accounting Account** directly on a product by going to the **Products** sidebar item, clicking **Edit** on a product row, and using the autocomplete account field.

**Confirms requirement:** The Allocation Rules page allows routing the same product to different accounts based on price component (e.g., WASTE_FEE to one account, TRANSPORT_FEE to another), matching the requirement's examples of VAT, waste treatment, and transport fees going to separate accounts.

**UI Status:** ✅ VISIBLE

### AC5 — Automatic completion used when entering account information

1. Click **Master Data** in the top navigation bar.
2. In the left sidebar, click **Allocation Rules** (or **Products**).
3. Click **+ Add Rule** (or **Edit** on a product row).
4. In the **Accounting Account** field, begin typing an account code or name (at least 2 characters).
5. A dropdown will appear below the field showing matching accounts with the account code in bold and the name alongside it.
6. Use the arrow keys or mouse to select a matching account from the dropdown.

**Confirms requirement:** The autocomplete input field searches accounts by code or name as the user types and presents matching suggestions in a dropdown, exactly as the requirement specifies.

**UI Status:** ✅ VISIBLE

### AC6 — Accounts managed in settings with code, name, validity period; all accounts visible to main users

1. Click **Master Data** in the top navigation bar.
2. In the left sidebar, click **Accounting**.
3. You will see a table showing all accounting accounts in the system with columns: Code, Name, Valid From, Valid To, and Created By.
4. Accounts with no end date display "Active" in the Valid To column.
5. To add an account, click **+ Add Account**. A modal opens with required fields: **Code**, **Name**, **Valid From**, and optional **Valid To**.
6. To edit an account, click **Edit** on any row. The same modal opens pre-filled with the account's current data.
7. To delete an account, click **Delete** on any row and confirm.
8. To filter accounts active on a specific date, use the **Filter by date** date picker at the top of the page. Click **Clear** to remove the filter.

**Confirms requirement:** All accounts in the system are visible in a single table. Each account has a code, name, valid-from date, and optional valid-to date. Creation and editing are available to users with access to Master Data.

**UI Status:** ✅ VISIBLE

### AC7 — Account information can be set for each individual pricing component of a product

1. Click **Master Data** in the top navigation bar.
2. In the left sidebar, click **Allocation Rules**.
3. Click **+ Add Rule**.
4. Select a **Product** from the dropdown.
5. In the **Price Component** dropdown, select the specific pricing component (e.g., WASTE FEE, TRANSPORT FEE, ECO FEE, SURCHARGE, or ADJUSTMENT).
6. In the **Accounting Account** field, type an account code or name and select a matching account from the autocomplete dropdown.
7. Optionally add region and municipality for more specific targeting.
8. Click **Save**. The new rule appears in the list.
9. Repeat with a different price component to route the same product's different fee types to different accounts.

**Confirms requirement:** The Allocation Rules form includes a Price Component field allowing account assignment to be targeted at each individual pricing component (waste fee, transport fee, etc.) of the same product.

**UI Status:** ✅ VISIBLE

### AC8 — Cost centers determined for events according to configuration rules

1. Click **Master Data** in the top navigation bar.
2. In the left sidebar, click **Cost Centers** to view and manage the composite cost center codes.
3. In the left sidebar, click **Products** to view or set a default cost center on each product via the **Edit** modal.
4. During invoice generation, the system automatically applies the matching cost center to each billing event's line items based on the configured rules.
5. The assigned cost center is visible on invoice line items (see AC9 flow).

**Confirms requirement:** Cost centers are configured in Master Data and the system automatically resolves them per event/line item according to product-level defaults.

**UI Status:** ✅ VISIBLE

### AC9 — Cost center and accounting account info exported as part of FINVOICE data

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice List**.
3. You will see a list of invoices. Click any invoice row to open the Invoice Detail page.
4. Scroll down to the **Line Items** section. The table shows columns including **Account** and **Cost Center** for each line item.
5. The account code and cost center composite code displayed here are the values that will be exported in the FINVOICE output for this invoice.

**Confirms requirement:** Every invoice line item shows the resolved accounting account code and cost center code, confirming this data is present in the invoicing data for FINVOICE export.

**UI Status:** ✅ VISIBLE

### AC15 — Same product routed to different account/cost center based on price list or region

1. Click **Master Data** in the top navigation bar.
2. In the left sidebar, click **Allocation Rules**.
3. You will see rules listed with columns for Product, Region, Municipality, Price Component, Account, and Specificity.
4. Click **+ Add Rule**.
5. Select the same product twice in two separate rules, each with a different region or municipality — and assign a different accounting account to each.
6. Click **Save** for each rule.
7. You can also test the routing using the **Resolve Test** panel at the bottom of the page: select the product, enter a region/municipality, and click **Resolve** to see which rule the system would select.

**Confirms requirement:** The Allocation Rules system explicitly supports routing the same product to different accounts based on region and municipality (which correspond to price list / geographic segments). The built-in Resolve Test panel lets users verify the routing logic interactively.

**UI Status:** ✅ VISIBLE

## Backend-Only Aspects

- **AC13 (Cash register system)** — Billing data originating from cash registers is ingested by the backend via an integration pipeline. There is no user-facing screen to configure this connection; it is set up during implementation at the infrastructure level.
- **AC14 (Weighing system)** — Weighbridge data is similarly processed by the backend. The `/integration/weighbridge-config` route exists in the router but is not surfaced in the sidebar navigation, so it is effectively not visible to end users at this time.
- **AC1 / AC8 (Automatic rule resolution)** — The actual matching of allocation rules and cost centers to events happens automatically in the backend at invoice generation time. Users configure the rules, but the runtime resolution is backend-only.
