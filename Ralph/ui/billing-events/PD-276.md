## Where to Find This Feature

This feature lives under the **Operations** tab. The main entry point is the **Billing Events** list, where individual event transfers (and bulk transfers) are initiated. The full audit trail and transfer chain are visible on the event detail page.

### AC1 — Uninvoiced events can be edited or reassigned before invoicing

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. You will see a table of billing events with columns for Date, Customer #, Product, fees, Status, and Validation.
4. Locate an event with status **IN_PROGRESS** (this means it has not been invoiced yet). Click the **Edit** button on that row (or click **View** and then **Edit** from the detail page header).
5. The edit form opens. You can change the Customer Number, Location, Product, fees, date, and other fields. Every change requires a mandatory reason in the **Reason for Edit** field at the bottom of the form.
6. Click **Save Changes**. The event returns to its detail page.

**Confirms requirement:** The system allows office staff to edit and reassign events line by line before they are invoiced, with a mandatory reason recorded for every change.

**UI Status:** ✅ VISIBLE

### AC3 — Transfer uninvoiced event from one customer to another

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. Find an event with status **IN_PROGRESS**. Click **View** on that row.
4. On the event detail page, you will see a **Transfer** button in the header (it is only visible when the event is IN_PROGRESS and not excluded).
5. Click **Transfer**. A modal dialog opens showing the current customer number.
6. In the **Target Customer** field, type the name, customer number, or address of the new customer. A live search dropdown appears — click the correct result to select it.
7. Optionally select a **Target Property** from the second search field.
8. Fill in the mandatory **Reason** text box.
9. Click **Next: Review →**. A confirmation screen shows the current customer → new customer arrow, the target property (if selected), and the reason.
10. Click **Initiate Transfer**. The event status changes to **Pending Transfer** and a banner appears at the top of the detail page showing "Transfer pending: [old customer] → [new customer]".
11. To finalise, click **Confirm Transfer** (or click **Cancel Transfer** to abort).

**Confirms requirement:** Users can move an uninvoiced event from its current customer to a different customer, with a two-step confirmation flow.

**UI Status:** ✅ VISIBLE

### AC4 — System records original customer; new customer billed correctly

1. After a transfer has been confirmed (AC3 above), navigate back to the event detail page.
2. On the **Details** tab, a yellow banner near the top reads "Transferred from: [original customer number]" and optionally "Prior property: [id]".
3. The event's Customer # field in the details grid now shows the new customer number.

**Confirms requirement:** The original customer is permanently displayed on the event, and the new customer's number is the active billing target.

**UI Status:** ✅ VISIBLE

### AC5 — Previous billing adjustments recorded for regulatory compliance

1. Open any billing event detail page (Operations → Billing Events → View).
2. Click the **Audit Trail** tab.
3. A table appears with columns: Timestamp, User, Field, Previous Value, New Value, and Reason.
4. Every transfer, edit, exclusion, and status change is listed. You can filter the log by user name, field changed, or date range.

**Confirms requirement:** A complete, tamper-evident record of all changes is visible, including who made each change, when, and why.

**UI Status:** ✅ VISIBLE

### AC7 — Move an incorrectly assigned event to the correct location before invoicing

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. Find an event with status **IN_PROGRESS**. Click **View**.
4. Click **Transfer** in the page header.
5. In the **Target Customer** field, select the same or a different customer.
6. In the **Target Property** field (which becomes available after a customer is selected), search for the correct address or property ID and select it. The property's street address and property ID are displayed in the dropdown.
7. Enter a **Reason** and click **Next: Review →**, then **Initiate Transfer**, then **Confirm Transfer**.
8. The event is now linked to the correct location. The prior location is shown in the yellow banner on the detail page.

**Confirms requirement:** An incorrectly assigned event can be moved to the correct property address before any invoice is generated.

**UI Status:** ✅ VISIBLE

### AC8 — Events can be corrected individually or in bulk while preserving service details

**Individual correction** is described in AC3/AC7 above.

**Bulk transfer:**

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. Use the checkboxes on the left of each row to select multiple events. Use the header checkbox to select all events on the current page.
4. In the toolbar above the table (below the filter bar), click **Transfer Selected**.
5. A "Transfer N Events" modal opens. Search and select the **Target Customer**, optionally a **Target Property**, and enter a mandatory **Reason**.
6. Click **Next: Review →**, then **Initiate Transfer for N Events**.
7. A green success message confirms the bulk transfer was initiated. All eligible events move to **Pending Transfer** status. SENT or COMPLETED events are automatically skipped.

**Confirms requirement:** Both individual and bulk transfers are supported, preserving all existing service detail fields while only changing the customer/location assignment.

**UI Status:** ✅ VISIBLE

### AC10 — Only the relevant service lines are moved; other charges remain unchanged

When the Transfer modal is used for a single event:

1. The transfer moves the entire event record to the new customer/location, but all fee components (Waste Fee, Transport Fee, Eco Fee), quantity, weight, product, and service details remain intact.
2. The Component Billing section on the detail page (checkboxes for Waste Fee, Transport Fee, Eco Fee) allows staff to toggle which fee components are included for invoicing independently of the transfer.

**Confirms requirement:** Only the customer/location assignment changes during a transfer; all service detail fields are preserved.

**UI Status:** ✅ VISIBLE

---

### AC11 — Prevents incorrect total amounts; only valid services reallocated

The Transfer modal moves the full event — it does not support selecting individual fee components to exclude from the transfer within the transfer flow itself. Fee component inclusion is managed separately in the Component Billing section of the detail page (toggling Waste Fee, Transport Fee, and Eco Fee checkboxes). There is no in-transfer filtering to restrict which fee lines are moved.

**Confirms requirement:** Partially covered — the transfer itself is precise at the event level, and Component Billing allows toggling fee components on/off, but these two actions are separate steps rather than integrated into the transfer flow.

**UI Status:** ⚠️ PARTIAL

**What is missing:** The transfer dialog has no option to select which fee components to include in the transfer. A user must separately go to the Component Billing section after the transfer to adjust which fees will be invoiced. The intent of the requirement (preventing incorrect totals during transfer) is achievable but requires two separate actions.

### AC13 — Only authorised personnel can perform transfers

1. The **Transfer** button on the event detail page is only rendered when the event status is exactly **IN_PROGRESS** and the event is not excluded (`canTransfer = !event.excluded && event.status === 'IN_PROGRESS'`).
2. The **Bulk Transfer** toolbar button on the Billing Events list is disabled unless at least one event is selected.
3. The underlying API calls (`/api/v1/billing-events/{id}/transfer`) are secured by role-based authentication on the backend. The Edit page additionally checks the user's role via the `/api/v1/me/roles` endpoint and restricts fields accordingly (INVOICING_PRICING role users see only pricing fields).

**Confirms requirement:** Transfer actions are gated behind both status conditions and backend role enforcement, ensuring only authorised users can initiate transfers.

**UI Status:** ✅ VISIBLE

---

### AC14 — Every event adjustment is logged with who made it, when, and why

1. Open any billing event detail page (Operations → Billing Events → View on any row).
2. Click the **Audit Trail** tab.
3. The audit log table shows every recorded change with columns: Timestamp, User (who made the change), Field (which data field changed), Previous Value, New Value, and Reason.
4. The reason field is mandatory for all edit and transfer operations, so every entry has a documented justification.
5. The log can be filtered by user name, field type, or date range to find specific changes.

**Confirms requirement:** A full change history is visible on every event, with timestamps, user identity, and mandatory reasons for every transfer and edit.

**UI Status:** ✅ VISIBLE
