## Where to Find This Feature

This feature is spread across three areas of the app. The core error list and per-event validation live under **Operations → Billing Events**. User-defined validation rules are configured under **Customers & Config → Validation Rules**. Annual billing threshold alerts (the "excessive container emptying" use case) are managed through the **Tools** tab via the Billing Threshold Alerts and Billing Threshold Config pages, which are accessible via direct link from those pages.

## User Flows by Acceptance Criterion

---

### AC1 — Mandatory data required before transfer to billing

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. You will see a table of billing events. Each row shows a **Validation** column displaying the event's validation status (a dash for unvalidated, "Passed", "Failed", or "Override").
4. Select one or more events by ticking the checkbox on the left of each row.
5. Click the **Validate** button in the toolbar above the table (the button shows the count of selected events).
6. A **Validation Report** modal will appear showing how many events were checked, how many passed, and a list of any **Blocking Failures** in red and **Warnings** in amber. Each failure shows the rule code, a plain-English description, and the specific field that failed.
7. Close the modal. Events with blocking failures will show "Failed" in their Validation column and cannot be transferred to invoicing until the failures are resolved.

**Confirms requirement:** The validation report modal lists every mandatory field failure (such as missing cost center or accounting account) and blocks the event from proceeding until the issue is corrected.

**UI Status:** ✅ VISIBLE

---

### AC2 — System supports user-defined automated checks

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Validation Rules**.
3. You will see a table of all configured validation rules, with columns for Rule Code, Type, Blocking, Active, and Description.
4. Click **+ Add Rule** in the top-right corner.
5. A modal opens. Choose a **Rule Type** from the dropdown — options include MANDATORY_FIELD, PRICE_CONSISTENCY, QUANTITY_THRESHOLD, CLASSIFICATION, REPORTING_DATA_COMPLETENESS, and VAT_ACCURACY.
6. Enter a **Rule Code** (e.g. BUSINESS_ID_MANDATORY), optionally add a **Config JSON** to parameterise the check (e.g. `{"field":"billingProfile.businessId"}`), a description, and mark whether the rule is **Blocking** (prevents invoice generation) and **Active**.
7. Click **Save**. The new rule appears in the table and will be applied automatically when billing events are validated.

**Confirms requirement:** Users can create custom rules of multiple types (mandatory fields, price consistency, quantity thresholds, VAT accuracy) with configurable parameters, satisfying the "case-by-case" user-defined checks the requirement describes.

**UI Status:** ✅ VISIBLE

### AC3 — Alert triggered for billing events exceeding configured thresholds (e.g. 30+ container emptying)

There are two complementary surfaces for this: configuring the threshold limit, and viewing triggered alerts.

**Configuring the threshold:**

The Billing Threshold Config page is accessible via a direct link (not currently in the main sidebar). Navigate to it by typing or linking to the admin config area.

1. Once on the **Billing Threshold Configuration** page, you will see any existing threshold rules with their Service Responsibility category and Euro limit.
2. Click **+ Add Threshold** to create a new rule.
3. Select a **Service Responsibility** (e.g. MUNICIPALITY), enter an **Annual Euro Limit**, and an optional description.
4. Click **Create**. The system will now automatically flag any customer whose annual billed amount exceeds this limit.

**Viewing triggered alerts:**

1. Click **Tools** in the top navigation bar.
2. In the left sidebar, click **Authority View** or navigate directly to the Billing Threshold Alerts page (accessible via a direct link from the threshold config area or PD Overview).
3. The **Billing Threshold Alerts** page has three tabs: **Triggers**, **Tickets**, and **Customers**.
4. The **Triggers** tab lists all customers whose annual billed amount has exceeded the configured threshold, showing the customer number, classification, annual amount (in red), threshold, year, and status.
5. Filter by Classification, Status, or Customer # and click **Search** to narrow results.
6. Click **View** on a trigger row to see the full detail page, including how much the threshold was exceeded by, a review decision form, and the customer's recent billing history.
7. From the trigger detail, a billing clerk can click **Record Review Decision**, enter a justification, and save — or click **Convert to Ticket** to create a structured case for follow-up.

**Confirms requirement:** The system automatically detects when a customer's billing exceeds the configured threshold and surfaces it in a filterable alert list, mirroring the use case of detecting 100 container emptyings when only 10 were expected.

**UI Status:** ✅ VISIBLE

### AC4 — Automatic validation before invoicing; error list produced

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. The stat strip at the top of the page shows a count for **Validation Failed** events. Click this tile to instantly filter the table to show only events that have failed validation.
4. You can also use the **Validation** filter dropdown in the filter bar — select "Failed" and click **Apply**.
5. Select all failed events using the checkbox in the table header.
6. Click **Validate** in the toolbar. A **Validation Report** modal appears listing every check that failed across the selected events, broken down into Blocking Failures and Warnings. Each failure shows the rule, description, and offending field.
7. Close the modal. The billing clerk now has a clear error list to work through before invoicing can proceed.

**Confirms requirement:** The Validation Report modal is the error list. It is produced automatically by the system when validation is run, and events with blocking failures are prevented from being transferred to invoicing.

**UI Status:** ✅ VISIBLE

### AC5 — Mandatory fields checked (customer info, pricing, VAT, accounting, service details)

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. Click the **View** button on any billing event row.
4. On the event detail page, scroll down past the main detail grid to find the **Validation Status** panel.
5. The panel shows the current validation status (Not yet validated / Passed / Failed / Overridden) and the date of the last validation run.
6. Click **Run Validation**. The system checks all configured rules, including mandatory fields for customer information, pricing, VAT classification, accounting account, cost center, and service-specific details.
7. If any mandatory fields are missing or incorrect, the status changes to **Failed** and a red panel appears listing each blocking failure — showing the rule type (e.g. MANDATORY_FIELD) and a description of what is missing.

**Confirms requirement:** The validation section on the event detail page shows which specific mandatory fields are missing or invalid, covering all the field categories listed in the requirement (customer info, VAT, accounting, service responsibility, etc.).

**UI Status:** ✅ VISIBLE

### AC6 — Configurable validation rules: price consistency checks and quantity thresholds

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Validation Rules**.
3. Click **+ Add Rule**.
4. To create a price consistency check: select Rule Type **PRICE_CONSISTENCY**, enter a rule code (e.g. PRICE_MATCHES_PRICELIST), add config JSON if needed, and mark it Blocking.
5. To create a quantity threshold: select Rule Type **QUANTITY_THRESHOLD**, enter a rule code (e.g. QUANTITY_EXCEEDS_LIMIT), and add config JSON with the threshold value.
6. Click **Save**. The rule is now active and will run automatically during the next validation pass on billing events.
7. Existing rules can be edited (click **Edit** on any row) or deleted (click **Delete**).

**Confirms requirement:** Users can define PRICE_CONSISTENCY and QUANTITY_THRESHOLD rule types with custom config, satisfying the requirement for user-built automations with user-defined conditions.

**UI Status:** ✅ VISIBLE

### AC7 — Error list reviewed by billing clerk; clerk corrects or overrides

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. Filter by **Validation Status: Failed** and click **Apply** to see the error list.
4. Click **View** on an event with a "Failed" validation status.
5. On the detail page, the **Validation Status** panel shows the list of failures in red (blocking) or amber (warnings), each with a plain-English description.
6. To correct the issue: click **Edit** (top-right button) to open the edit form, fix the flagged fields (e.g. assign the missing cost center or accounting account), and save.
7. To override without correcting: in the Validation Status panel, click **Override Validation**, enter a documented reason explaining why proceeding despite the failures is acceptable, and click **Confirm Override**. The status changes to "Override" with the reason and the user's name recorded.

**Confirms requirement:** Billing clerks can see exact failure details per event, correct the data by editing the event, or override the validation check with a documented reason — exactly as the requirement describes.

**UI Status:** ✅ VISIBLE

---

### AC8 — Events blocked from invoicing until all mandatory fields are valid

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. Select one or more events that have a "Failed" validation status.
4. Click **Transfer Selected** in the toolbar.
5. The system will block the transfer for events with blocking validation failures. Events with unresolved blocking failures cannot be moved to the invoicing pipeline.
6. Events must be either corrected (so validation passes) or the validation must be overridden with a reason before they can proceed to invoicing.

**Confirms requirement:** The transfer action is gated on validation status — events with blocking failures are excluded from the invoicing pipeline until the failures are resolved or overridden.

**UI Status:** ✅ VISIBLE

---

### AC9 — Audit and compliance tracking of all validation results and corrections

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. Click **View** on any billing event.
4. Click the **Audit Trail** tab at the top of the event detail page.
5. The Audit Trail tab shows a chronological log of all actions taken on the event, including validation runs, corrections, status transitions, overrides, and who performed each action and when.
6. If the event had a validation override, the **Validation Status** panel on the Details tab also records the override reason, the user who approved it, and the timestamp.

**Confirms requirement:** The Audit Trail tab provides full traceability for validation results, corrections, and overrides — satisfying the audit and compliance tracking requirement.

**UI Status:** ✅ VISIBLE
