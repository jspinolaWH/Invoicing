## Where to Find This Feature

This feature lives under the **Operations** tab. Most of it is accessed through the Billing Events list, where users can select and transfer one or many uninvoiced events at once, and from the Billing Event Detail page, which also handles transfer confirmation and correction of already-invoiced events via Credit & Transfer.

## User Flows by Acceptance Criterion

---

### AC1 — Transfer a single uninvoiced event to a new customer

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. You will see a table of billing events. Each row shows the date, customer number, product, fees, status badge, and an **Actions** column.
4. Find an event with status **In Progress** (events in DRAFT status cannot be transferred; only IN_PROGRESS events show the Transfer button). Click **View** on that row.
5. You land on the Billing Event Detail page. At the top you will see the event number, a status badge, and a row of action buttons.
6. Click the **Transfer** button (visible only when the event's status is IN_PROGRESS and it is not already excluded).
7. A modal titled **Transfer Billing Event** opens. It shows the current customer number for reference.
8. In the **Target Customer** field, type part of the new customer's name, number, or address. A dropdown of matching customers appears; click the correct one. The customer number is confirmed below the field.
9. Optionally, use the **Target Property** field to search for a specific property belonging to that customer.
10. In the **Reason** field, type the reason for the transfer (required).
11. Click **Next: Review →**.
12. A review summary is shown: current customer number, an arrow, the new customer number, optional property, and the reason you entered.
13. Read the note: the event will move to **Pending Transfer** state and requires a second confirmation.
14. Click **Initiate Transfer**.
15. The modal closes. The event's status badge at the top of the detail page changes to **Pending Transfer**. A blue banner appears showing "Transfer pending: [old customer] → [new customer]".

**Confirms requirement:** The user can move any IN_PROGRESS billing event to a different customer. The event enters Pending Transfer state, providing a two-step safety check before the reassignment is finalised.

**UI Status:** ✅ VISIBLE

### AC2 — Transfer events from shared services or waste reception

The transfer modal (TransferEventModal) and the bulk transfer modal (BulkTransferModal) do not filter by event origin — they work on any billing event regardless of whether it originated from a shared service collection group, waste reception, or another source. The **Billing Events** list displays an **Origin** column (values such as MANUAL, SHARED_SERVICE, DRIVER, etc.) and the same **Transfer** button path is available on the detail page for any IN_PROGRESS event of any origin type.

However, the UI does not provide a specific filter for "shared service" or "waste reception" origin in the list to make it easy to find only those event types for bulk transfer. Users must identify them manually using the Origin column or the Service Responsibility filter.

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. In the filter bar, enter a value in the **Service Responsibility** field to narrow results, or look at the **Origin** column in the table to identify shared service or waste reception events.
4. Select the relevant rows using the checkboxes on the left. For a single event, click **View** to open the detail page.
5. For a single event on the detail page: click **Transfer** and follow the same steps as AC1.
6. For multiple events selected from the list: click **Transfer Selected** in the toolbar above the table. The **Transfer [n] Events** bulk modal opens. Fill in the target customer, optionally a target property, and a reason. Click **Next: Review →**, then **Initiate Transfer for [n] Events**.
7. All eligible events move to Pending Transfer. A note in the modal warns that SENT or COMPLETED events are skipped automatically.

**Confirms requirement:** Events from any origin (including shared services and waste reception) can be transferred individually or in bulk using the same UI flows.

**UI Status:** ⚠️ PARTIAL

There is no dedicated filter to isolate shared-service or waste-reception events as a group before bulk-transferring them. Users must use the general Origin column or the Service Responsibility text filter to identify them manually. The transfer action itself works correctly for all origins.

### AC3 — Understand what a billing event is (contextual information)

The Billing Events list page provides a clear title "Billing Events" and a subtitle "Manage and validate billing events before invoicing." Each event row shows the date, customer number, product name, fees (waste fee, transport fee, eco fee), quantity, and status — giving users enough context to understand that an event represents a single chargeable action such as a container emptying or a weighbridge weighing.

The detail page shows further fields: Origin badge (DRIVER, MANUAL, SHARED_SERVICE, etc.), weight, quantity, location, vehicle, driver, and contractor — together these describe what triggered the event.

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. You will see the list with columns: Date, Customer #, Product, Waste Fee, Transport Fee, Eco Fee, Quantity, Classification, Status, Validation, Origin.
4. Click **View** on any event row to open the detail page and see the full set of fields describing the event.

**Confirms requirement:** The list and detail pages together make it clear that a billing event represents a specific chargeable action (emptying, weighbridge weighing, etc.) that is to be invoiced to a customer.

**UI Status:** ✅ VISIBLE

### AC4 — Find uninvoiced events for a wrong customer and transfer them to the new resident

This flow covers the scenario: a resident has moved out, the invoicing user spots new IN_PROGRESS events still assigned to the old customer, and transfers them all to the new resident.

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. In the filter bar, type the old customer's number in the **Customer #** field and choose **In Progress** in the status dropdown. Click **Apply**.
4. The table now shows only IN_PROGRESS events for that customer. Review the list to confirm these are the uninvoiced events that should go to the new resident.
5. Select all relevant rows by ticking each checkbox (or use the header checkbox to select all events on the current page).
6. In the toolbar above the table, click **Transfer Selected**.
7. The **Transfer [n] Events** modal opens. In **Target Customer**, search for and select the new resident.
8. Optionally set a **Target Property** (e.g. the new apartment's address).
9. Enter a **Reason** (e.g. "Resident moved — events reassigned to new occupant").
10. Click **Next: Review →** to see the summary, then click **Initiate Transfer for [n] Events**.
11. A green success message appears: "Bulk transfer completed." The transferred events now appear in the list with status **Pending Transfer**.
12. To finalise each transfer, click **View** on a Pending Transfer event, then click **Confirm Transfer** on the detail page.

**Confirms requirement:** The user can locate all uninvoiced (IN_PROGRESS) events for the wrong customer using filters, select them in bulk, and transfer them to the correct new resident in a single workflow. The two-step confirmation (initiate then confirm) ensures the change is deliberate.

**UI Status:** ✅ VISIBLE

### AC5 — Correct the invoiced events (already sent/completed)

When an event has already been invoiced (status SENT or COMPLETED), a simple transfer is not available. Instead the user uses the **Credit & Transfer** action, which issues a credit against the original invoice and creates a new event for the correct customer.

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. Filter by the old customer's number and choose status **Sent** or **Completed**. Click **Apply**.
4. Click **View** on the already-invoiced event you need to correct.
5. On the Billing Event Detail page, click **Credit & Transfer** (visible only when the event is in SENT or COMPLETED status and not already part of a credit-transfer chain).
6. A modal titled **Credit & Transfer Billing Event** opens. It shows a warning banner explaining that a credit will be issued for the original invoice and a new event created for the target customer. The original event will not be modified.
7. Below the warning, a summary of the original event is shown: event number, customer, product, and total fees.
8. In **Target Customer**, search for and select the new (correct) customer.
9. Optionally choose a **Target Property**.
10. Enter a **Reason** (required; appears in the audit trail).
11. Click **Confirm Credit & Transfer**.
12. The modal closes. The original event's detail page now shows a blue banner: "A credit & transfer was issued for this event." Two buttons appear: **View Credit Event** and **View Transfer Event**.
13. On the original event, a **Credit & Transfer** button is no longer shown (preventing duplicate corrections).
14. Alternatively, the user can correct an invoice at the invoice level: navigate to **Invoices** in the top nav → **Invoice List**, find and click the invoice, then click **Correct** to open the Correct Invoice form. This form allows selecting a transfer mode or copy mode, targeting a different customer, and choosing which billing event lines to include in the credit.

**Confirms requirement:** The user can reverse the billing of an already-invoiced event and redirect the charge to the correct customer via Credit & Transfer, satisfying the requirement that invoiced events be correctable.

**UI Status:** ✅ VISIBLE
