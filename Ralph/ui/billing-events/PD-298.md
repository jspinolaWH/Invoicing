## Where to Find This Feature

This feature spans two main areas of the app. Customer billing data (identifiers, delivery method, address) is managed under the **Customers & Config** tab on the **Billing Profiles** page. Validation of that data before invoice generation and the transfer of invoice data to external systems is handled under the **Invoices** tab — through the **Generate Invoice** page and individual **Invoice Detail** pages.

### AC1 — System extracts invoice delivery information from customer profiles

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. You will see a table of all customers. Click any customer row to open their billing profile.
4. The billing profile page loads automatically, showing the stored delivery method, billing address, Customer ID Number, Business ID, e-invoice address, and other invoice-related fields — all drawn directly from the customer record.

**Confirms requirement:** The billing profile page shows all invoice-delivery-related fields drawn automatically from the stored customer record, confirming that the system extracts this data from customer profiles.

**UI Status:** ✅ VISIBLE

### AC2 — Mandatory data fields: customer identifiers, delivery method, and billing address

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. Click any customer row to open their billing profile.
4. On the **Profile** tab, you will see:
   - A **Customer ID Number** field marked with a red asterisk (required), showing the 6–9 digit identifier.
   - A **Delivery Method** dropdown marked as required, with options: E INVOICE, EMAIL, PAPER, DIRECT PAYMENT.
   - A **Billing Address** section with required fields for Street Address, Postal Code, City, and Country Code — all marked with red asterisks.

**Confirms requirement:** All three mandatory data categories — customer identifier, delivery method, and billing address — are present and marked as required fields on the billing profile form.

**UI Status:** ✅ VISIBLE

### AC4 — Invoice delivery method is recorded (e-invoice, email, paper)

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. Click any customer row to open their billing profile.
4. On the **Profile** tab, find the **Delivery Method** dropdown.
5. The dropdown offers four options: **E INVOICE**, **EMAIL**, **PAPER**, and **DIRECT PAYMENT**.
6. When **E INVOICE** is selected, an additional **E-Invoice Address** section appears below (after saving), where the e-invoice address and operator code can be entered.
7. A warning message appears if you switch away from E INVOICE when an e-invoice address is already stored.

**Confirms requirement:** The delivery method dropdown records exactly how the invoice will be sent (e-invoice, email, paper, or direct payment), and the choice persists in the customer record.

**UI Status:** ✅ VISIBLE

### AC5 — System prevents invoice finalisation when critical customer data is missing

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Generate Invoice**.
3. Enter a Customer ID, a billing period start date, and a billing period end date.
4. Click **Preview Invoice**.
5. The system runs a validation against the customer's profile. If critical data is missing (for example, Business ID for e-invoice customers, or missing billing address fields), a list of **blocking validation failures** appears in the preview panel, shown in red.
6. When blocking failures exist, the **Confirm & Generate** button is disabled with a message reading "[n] blocking validation failure(s) — cannot generate invoice."
7. The user cannot proceed until the missing data has been corrected in the customer's billing profile.

Alternatively, from a customer's billing profile page:

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**, then click a customer row.
3. Click the **Validate Profile** button (top right of the page).
4. A **Validation Report** modal opens showing the total number of checks, how many passed, any blocking failures (in red), and any warnings (in yellow), each with a description and the specific field that failed.

**Confirms requirement:** The system actively blocks invoice generation when critical customer data is missing, and the **Validate Profile** function gives users an early view of data quality issues.

**UI Status:** ✅ VISIBLE

---

### AC6 — Notifies users of invalid data and lists issues to correct before continuing

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Generate Invoice**.
3. Enter a Customer ID and billing period dates, then click **Preview Invoice**.
4. If there are validation issues, the preview panel shows a colour-coded list:
   - Red boxes for **blocking** failures — each shows the rule type and a plain-English description of the problem.
   - Yellow boxes for **warnings** — same format but advisory only.
5. The user reads the listed issues and navigates to the customer's billing profile to correct the missing or invalid data, then re-previews.

Alternatively, on the billing profile page, clicking **Validate Profile** opens a **Validation Report** modal that lists every blocking failure and warning with the rule name, description, and the specific field involved.

**Confirms requirement:** Both the Generate Invoice preview and the Validate Profile modal explicitly list each data issue so the user knows exactly what to fix before proceeding.

**UI Status:** ✅ VISIBLE

### AC7 — Automatic extraction of customer data from customer records during invoice generation

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Generate Invoice**.
3. Enter the **Customer ID** and billing period dates, then click **Preview Invoice**.
4. The preview panel shows invoice totals and a sample of line items automatically populated from the customer's billing profile — including the customer name, amounts, and invoice structure — without the user needing to re-enter any customer data.

**Confirms requirement:** The invoice preview is built automatically from the stored customer record; the user only needs to provide the customer ID and period, confirming automatic extraction of customer data.

**UI Status:** ✅ VISIBLE

### AC8 — Invoice data file must comply with the FINVOICE 3.0 standard

There is no user-facing screen for FINVOICE 3.0 compliance configuration. This is enforced in the backend when generating the invoice data file that is transmitted to the operator. The user does not configure or review FINVOICE field mappings in the UI — the system produces a compliant file automatically.

**Confirms requirement:** Not directly visible — the FINVOICE 3.0 structure is applied by the backend engine at transmission time.

**UI Status:** ❌ NOT VISIBLE

*No frontend surface for FINVOICE 3.0 compliance — it is implemented in the backend invoice file generation and transmission layer.*

### AC9 — Validation and error prevention before invoice finalisation

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Generate Invoice**.
3. Enter a Customer ID and billing period, then click **Preview Invoice**.
4. The preview panel shows any validation failures. The **Confirm & Generate** button is disabled when blocking failures are present, preventing the user from finalising a bad invoice.
5. Alternatively, on the **Billing Profile** page (accessible via **Customers & Config** → **Billing Profiles** → click a customer row), click **Validate Profile** to trigger pre-invoice validation at any time.

**Confirms requirement:** The system prevents invoice finalisation when blocking validation failures exist, and provides a dedicated **Validate Profile** button for proactive validation.

**UI Status:** ✅ VISIBLE

### AC10 — Transfer of invoicing data to external invoicing systems

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice List**.
3. Click any invoice row to open the invoice detail page.
4. Scroll down to the **Transmission & Integration** section.
5. If the invoice status is **READY**, a **Transmit to Operator** button is available. Click it to send the invoice data to the external invoicing system.
6. After transmitting, the panel shows a confirmation with the external reference number, status, and timestamp.
7. With a **SENT** invoice, additional buttons appear: **Check Delivery Status**, **View Invoice Image (PDF)**, **Fetch External Attachments**, and **Retransmit (Updated Address)**.
8. If the customer profile has external recall enabled, a **Recall Invoice** button also appears, allowing the invoice to be withdrawn from the operator system.

**Confirms requirement:** The Transmission & Integration panel on the invoice detail page provides explicit controls for sending invoice data to the external invoicing system and checking delivery status.

**UI Status:** ✅ VISIBLE

### AC12 — Basic FINVOICE data common for all companies; deviations handled via change request

There is no user-facing screen for FINVOICE baseline field configuration or for requesting deviations from the standard. This is an operational/contractual process handled outside the application (via a formal change request to WasteHero), with the resulting configuration applied in the backend.

**Confirms requirement:** Not visible — there is no UI for managing the FINVOICE standard baseline or logging deviations. This is handled operationally.

**UI Status:** ❌ NOT VISIBLE

*No frontend surface found. FINVOICE baseline configuration and company-specific deviations are managed as backend change requests, not through the UI.*

## Backend-Only Aspects

- **AC8 (FINVOICE 3.0 compliance):** The backend automatically structures the invoice data file according to the FINVOICE 3.0 standard when transmitting to the operator. No user action or configuration is required — compliance is guaranteed by the invoice generation engine.
- **AC12 (FINVOICE baseline / deviations via change request):** The definition of which FINVOICE fields are standard across all companies, and any company-specific deviations, is managed entirely through backend configuration and formal change requests. There is no UI for end users or administrators to modify this.
