## Where to Find This Feature

This feature lives under the **Operations** tab. Billing cycle configuration is managed through the **Billing Cycles** sidebar item, where office users can create, view, and edit schedules per customer, contract, and property. Invoice preview before finalization is accessible through the **Invoices** tab via **Invoice Runs** using the "Simulate First" mode.

## User Flows by Acceptance Criterion

---

### AC1 — Link services to a predefined billing cycle and adjust intervals

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Cycles**.
3. You will see a table listing all billing cycles with columns for Customer Number, Frequency, Next Billing Date, Service Type, Description, and Active status.
4. To create a new cycle, click **+ Add Cycle** in the top right corner.
5. A modal dialog appears. Fill in the Customer Number (required), then choose a Frequency from the dropdown — options are MONTHLY, QUARTERLY, SEMI_ANNUAL, and ANNUAL.
6. Set the Next Billing Date using the date picker. Optionally fill in Service Type, Contract Reference, Property Reference, and Description.
7. Ensure the **Active** checkbox is checked, then click **Save**.
8. The modal closes and the new cycle appears in the table.
9. To edit an existing cycle, click the **Edit** button on any row. The same modal opens pre-filled with the existing values. Change the Frequency or other fields and click **Save**.
10. A green success message ("Cycle created." or "Cycle updated.") appears at the top of the page.

**Confirms requirement:** The user can create and modify billing intervals (MONTHLY, QUARTERLY, SEMI_ANNUAL, ANNUAL) for any customer, with the change immediately reflected in the table.

**UI Status:** ✅ VISIBLE

---

### AC2 — Events grouped by billing period and validated before invoice generation

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Cycles** to verify the customer's cycle is configured.
3. Click **Invoices** in the top navigation bar.
4. In the left sidebar, click **Invoice Runs**.
5. Select the **Simulate First** radio button.
6. Configure run filters (customer number, date range, etc.) in the filter form.
7. Click the **Submit** button (or equivalent run action button).
8. The system runs a simulation and takes you to the Simulation Results page, where you can see Total Customers, Total Invoices, validation issues (BLOCKING or WARNING), and a breakdown of billed events by category.
9. Validation failures are shown with severity, customer name, rule type, and description — blocking failures prevent the real run.

**Confirms requirement:** Events are validated before invoices are generated (blocking failures halt the run), and the simulation shows grouping by category. However, there is no explicit per-billing-period grouping view showing which cycle each event belongs to.

**UI Status:** ⚠️ PARTIAL

**What is missing:** The simulation results page does not display events grouped explicitly by their assigned billing period (e.g., "Q1 events for Customer X"). The grouping happens in the backend but is not surfaced as a per-cycle breakdown in the UI.

### AC3 — Automatic billing cycle assignment (MONTHLY, QUARTERLY, SEMI_ANNUAL, ANNUAL)

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Cycles**.
3. Click **+ Add Cycle**.
4. In the modal, open the **Frequency** dropdown — you will see four options: MONTHLY, QUARTERLY, SEMI_ANNUAL, ANNUAL.
5. Select the desired frequency, complete the remaining fields, and click **Save**.

**Confirms requirement:** The frequency dropdown exposes all four named billing intervals, confirming that each service or event type can be assigned a specific billing schedule.

**UI Status:** ✅ VISIBLE

### AC4 — Independent service-based billing (different services, different schedules)

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Cycles**.
3. You will see rows in the table — each row represents one billing cycle record. Multiple rows can share the same Customer Number but have different Frequency values and different Service Type values.
4. To add a second cycle for the same customer with a different frequency, click **+ Add Cycle**, enter the same Customer Number, choose a different Frequency (e.g., QUARTERLY), and enter a different Service Type (e.g., CONTAINER_EMPTYING).
5. After saving, both cycles appear in the table for the same customer, each with its own schedule.
6. Use the **Filter by customer** input at the top of the page to type a customer number and view only that customer's cycles.

**Confirms requirement:** Multiple billing cycles can coexist for the same customer number with different frequencies, service types, property references, and billing dates, supporting fully independent service-level scheduling.

**UI Status:** ✅ VISIBLE

### AC5 — Automatic event grouping by schedule

This criterion describes backend behaviour: when an invoice run executes, the system automatically routes each billing event to the correct invoice according to the event's assigned billing cycle. There is no separate screen where the user manually triggers this grouping — it happens automatically as part of the run.

The user can observe the result in the Invoice Run Detail page (reached after committing a run from Invoice Runs), where total invoices produced are shown, and in the Simulation Results page where events are broken down by category.

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice Runs**.
3. Configure a run and click to start (real or simulated).
4. Review the resulting invoice count and category breakdown to confirm events were grouped automatically.

**Confirms requirement:** Grouping is automatic and visible in aggregate on the simulation/run results page, but there is no explicit per-cycle grouping report.

**UI Status:** ⚠️ PARTIAL

**What is missing:** No dedicated UI shows which events were grouped into which billing cycle's invoice. The grouping is backend-driven and confirmed only via the aggregate run summary.

### AC6 — Flexible user configuration of service- or property-specific billing cycles

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Cycles**.
3. Find the cycle you want to modify and click **Edit** on its row.
4. In the modal, you can change the Frequency, Next Billing Date, Service Type, Contract Reference, Property Reference, and Description independently — without affecting other cycles for the same customer.
5. Click **Save** to apply the change.

**Confirms requirement:** Each billing cycle record is independently editable. Changing one cycle's frequency or property reference does not alter any other cycle, confirming isolated configuration per service or property.

**UI Status:** ✅ VISIBLE

### AC7 — Invoice preview and confirmation before finalization

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice Runs**.
3. Select the **Simulate First** radio button at the top of the page.
4. Configure the run filters (date range, customer filters, etc.) using the filter form.
5. Click **Submit** (or the run action button).
6. The system runs a dry-run simulation and takes you to the **Simulation Results** page.
7. You will see summary cards showing Total Customers, Total Invoices, Failed Customers, Net Total, VAT Total, and Gross Total.
8. If there are validation issues, a table lists each issue with its Severity (BLOCKING or WARNING), Customer, Rule Type, and Description.
9. A "Sample Invoices (first 5)" section shows expandable entries for individual customers, including gross amount and line item count.
10. If there are no blocking failures, click **Commit Real Run** to finalize. If blocking failures exist, the button is disabled and you must resolve the issues first.

**Confirms requirement:** Users can review a full simulation of upcoming invoices — including totals, validation failures, and sample invoice breakdowns — before committing to a real run. Blocking failures prevent finalization until resolved.

**UI Status:** ✅ VISIBLE

### AC8 — Seamless integration with external financial and accounting systems

This criterion is satisfied entirely in the backend. When billing cycles are processed and invoices are generated, the system automatically synchronises the resulting invoice data with external financial and accounting systems via the existing integration layer. There is no user-facing screen that lets users manually trigger or configure this synchronisation specifically for billing cycles — it is part of the automated invoice run pipeline.

**UI Status:** ❌ NOT VISIBLE

**Why no UI:** The accounting/ERP integration is handled automatically when a real invoice run completes. The integration configuration (e-invoice operators, accounting accounts, VAT rates, etc.) is managed in Master Data and the Tools > E-Invoice Op. section, not from the Billing Cycles page.

### AC9 — Scenario: customer with two properties on different billing cycles

3. Click **+ Add Cycle** and create a cycle for the customer with:
   - Frequency: QUARTERLY
   - Property Reference: the residential property identifier
   - Service Type: e.g. MIXED_WASTE
   - Next Billing Date: the next quarter start

   - Frequency: ANNUAL
   - Property Reference: the summer cottage identifier
   - Service Type: e.g. WASTE_MANAGEMENT
   - Next Billing Date: e.g. April 1
5. After saving, filter by the customer number to see both cycles listed side by side.

**Confirms requirement:** The system supports creating two distinct billing cycles for the same customer number, each with its own property reference and frequency, matching the scenario described in the requirement.

### AC12 — ERP consolidates summer cottage events into single annual invoice in December

This criterion describes the automatic invoice generation behaviour when an annual billing run is triggered. The consolidation (all events for the summer cottage rolled into one annual invoice) is performed by the backend during the invoice run.

The user can observe the result by:

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice Runs**.
3. Run a simulation (select **Simulate First**) with a date range covering the annual billing period.
4. On the Simulation Results page, review the total invoices generated — one invoice per customer per billing period should be visible in the sample invoices section.
5. To commit, click **Commit Real Run**.

**Confirms requirement:** The consolidation logic runs automatically. The UI confirms it by showing one invoice entry per customer in the simulation results, but does not explicitly label that invoice as "annual" or show which billing cycle it originated from.

**UI Status:** ⚠️ PARTIAL

**What is missing:** The simulation results and run detail pages do not display per-cycle attribution (e.g., "this invoice was generated from the ANNUAL cycle for property X"). The user must infer consolidation occurred from the invoice count.
