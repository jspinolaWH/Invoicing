## Where to Find This Feature

This feature lives under the **Tools** tab. The E-Invoice Operator Management page allows manual triggering of the batch. The resulting changes to customers (billing channel and e-invoice address) are visible on each customer's Billing Profile page, reachable under **Customers & Config**.

### AC3 — Billing channel changed to e-invoice after START order

The integration automatically updates the customer's delivery method to E-INVOICE when a START order arrives. The result is visible on the customer's billing profile.

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. You will see a table of customers. Click any customer row to open their billing profile.
4. On the billing profile page, look at the **Delivery Method** field under "Billing Profile". If an e-invoice START order was processed for this customer, the value will show **E INVOICE**.
5. To confirm the change was made by integration (not manually), click the **Change History** tab at the top of the profile page.
6. You will see a table of changes. Look for a row where the **Field** column is `deliveryMethod`, the **New Value** is `E_INVOICE`, and the **Changed By** column shows `integration`.

**Confirms requirement:** The Change History tab shows that the delivery method was updated to E_INVOICE by the "integration" actor, proving the integration set the billing channel.

**UI Status:** ✅ VISIBLE

---

### AC4 — E-invoice address added for the customer

When a START order is processed, the customer's e-invoice address is populated. This is visible on the billing profile page.

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. Click the customer row for the customer who received the e-invoice order.
4. On the billing profile page, ensure the **Profile** tab is selected.
5. Scroll down past the main Billing Profile fields. Because the Delivery Method is now set to **E INVOICE**, a section labelled **E-Invoice Address** will be visible below the Save button.
6. In that section you will see the **E-Invoice Address** field and the **Operator Code** field, pre-filled with the values received from the integration.

**Confirms requirement:** The E-Invoice Address section is visible and populated with the address sent by the external billing intermediary.

**UI Status:** ✅ VISIBLE

### AC5 — The change (e-invoice start) is recorded

The audit trail for the e-invoice address update is stored and visible in the customer's Change History.

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. Click the relevant customer row.
4. Click the **Change History** tab on the billing profile page.
5. You will see a table of recorded changes. Look for a row where the **Field** column is `einvoiceAddress`, the **New Value** shows the new e-invoice address, and **Changed By** shows `integration`.

**Confirms requirement:** The Change History table records both the delivery method change and the e-invoice address update, attributed to the "integration" system user.

**UI Status:** ✅ VISIBLE

### AC6 — Billing channel changed to paper invoicing after TERMINATE order

When a TERMINATE order is received, the integration reverts the customer's delivery method to PAPER. This is visible on the billing profile.

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. Click the customer row for the customer whose e-invoice was terminated.
4. On the billing profile page, look at the **Delivery Method** field. It will show **PAPER**.
5. Click the **Change History** tab to confirm. Look for a row where the **Field** is `deliveryMethod`, **New Value** is `PAPER`, and **Changed By** is `integration`.

**Confirms requirement:** The Delivery Method field shows PAPER and the Change History confirms the integration made this change.

**UI Status:** ✅ VISIBLE

### AC7 — E-invoice information removed after TERMINATE order

When termination occurs, the e-invoice address is cleared. This is visible on the profile page.

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. Click the relevant customer row.
4. On the **Profile** tab, because the Delivery Method is now PAPER, the **E-Invoice Address** section will no longer be shown on the page (it only appears when the delivery method is E INVOICE).
5. To see the cleared address in the audit, click the **Change History** tab and look for a row where **Field** is `einvoiceAddress`, **New Value** is empty or blank, and **Changed By** is `integration`.

**Confirms requirement:** The E-Invoice Address section disappears from the profile when delivery method is PAPER, and the Change History records the address being removed.

**UI Status:** ✅ VISIBLE

The termination event is audited in the same Change History as the start event.

### AC9 — Data retrieval from billing system once per day

The requirement states the recommended approach is that changes are delivered once per day. The frontend provides a manual trigger page and indicates the batch runs automatically at 02:00, but there is no display of the scheduled run time or a configurable schedule setting.

1. Click **Tools** in the top navigation bar.
2. In the left sidebar, click **E-Invoice Op.**
3. You will see the **E-Invoice Operator Management** page. It states: "The daily batch runs automatically at 02:00. Use the button below to trigger it manually."
4. The page shows a **Trigger Batch Now** button.
5. Click **Trigger Batch Now** — a confirmation prompt appears.
6. Click **Confirm Run** to execute the batch immediately.
7. When complete, a green summary panel shows counts of **Started**, **Terminated**, and **Failed** registrations.

**Confirms requirement:** The page confirms a daily automatic schedule at 02:00 and allows manual triggering. However, there is no UI to view or configure the schedule itself, nor a log of previous automatic runs.

**UI Status:** ⚠️ PARTIAL

The daily batch time (02:00) is hard-coded in the backend and only mentioned as static text on the page. There is no configurable schedule, no history of automatic batch runs, and no display of which orders were processed in past runs. The manual trigger works and shows a result summary for that single run only.
