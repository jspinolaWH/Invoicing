## Recommendations

### Requirements needing UI attention (PARTIAL or NO_UI)

**NO_UI requirements (need frontend work):**
- **PD-427 — Invoicing REST API:** This is a pure backend/API specification requirement with no user-facing screen. A documentation or API explorer page could be considered but is not strictly required.
- **PD-163 — 3.7.15 Billing list:** The billing list/report feature has no dedicated frontend page. The existing Billing Events page partially addresses this, but a dedicated exportable billing list report page is missing.

**PARTIAL_UI requirements (most impactful gaps):**
- **PD-310 — FINVOICE data:** The invoice run triggers FINVOICE generation but users cannot inspect or validate the FINVOICE output in the UI before transmission.
- **PD-308 — Invoice language selection:** Language codes are not currently configurable on the customer billing profile page; the setting exists in the backend but is not surfaced in the UI.
- **PD-307 — Invoice template selection:** Template codes can be set during invoice runs but cannot be set per-customer or per-property in their respective profile pages.
- **PD-306/303 — Invoice image display:** There is no in-app invoice image viewer; users are redirected externally. The integration API hook exists but the display window is not implemented.
- **PD-305/304 — PDF attachments:** Users cannot attach PDFs to invoices or invoice batches from the UI; this is backend-only currently.
- **PD-302 — Custom invoice texts:** There is no free-text field on the invoice generation form for adding per-invoice or bulk custom messages.
- **PD-301 — Gross/net invoicing:** The gross/net setting is not visible or configurable on the customer billing profile.
- **PD-298 — Billing data details:** Some mandatory customer data validation fields (e-invoice address, Business ID) are not surfaced in the UI as blocking validation errors.
- **PD-295 — Account and cost center data:** Account autocomplete on product pricing components is present but the full cost center assignment UI is incomplete.
- **PD-293 — Invoice batch filtering:** The invoice run form has some filters but is missing several selection criteria (reception location, service responsibility filter, monetary threshold carry-over).
- **PD-292 — Billing restrictions:** Billing Restrictions page exists but lacks immediate-invoicing vs. scheduled rules per service type.
- **PD-291 — Billing cycles:** Billing Cycles page exists but per-property overrides and invoice preview before finalization are missing.
- **PD-285/284 — Public/private law:** Public vs. private law classification is recorded at the billing event level but is not filterable or reportable from the UI in a dedicated view.
- **PD-282 — Editing e-invoice data:** E-invoice operator/address fields are editable via the integration operator page but not from the customer billing profile directly.
- **PD-281 — Updating billing information via API:** This is largely a backend concern but the UI does not expose a webhook/API key management screen for external systems.
- **PD-276 — Transfer of unbilled events:** The transfer action exists on the billing events list but only for single events; bulk transfer is missing.
- **PD-274 — Invoice simulation:** Simulation results page exists but pre-run simulation filtering and threshold preview are limited.
- **PD-270 — System behavior during billing run:** The invoice run detail page shows status but does not expose pause/resume controls or per-event error handling during an active run.
- **PD-269 — Credit invoices:** Credit note form exists but lacks automatic credit matching to original invoice and bulk credit generation.
- **PD-177 — Billing event data for reporting:** The accounting report page exists but billing-event-level reporting fields (section 3.6 custom fields) are not filterable in that view.
- **PD-107 — E-invoice integration:** The E-Invoice Operator page exists but lacks status monitoring, retry controls, and transmission history for sent e-invoices.
- **PD-344 — Transfer of billing events:** Single event transfer exists; transfer of shared-service events and bulk transfer are not fully implemented.
- **PD-363 — Euro limit alerts:** Admin threshold config and alert list pages exist but the ticket conversion workflow (AC4–6) and full audit trail view (AC7) are not implemented.
- **PD-364 — Retroactive service responsibility changes:** The Responsibility Change page exists for applying changes, but the audit log of who changed what and the reporting toggle (original vs. updated) are not surfaced.

### Orphaned pages needing requirements

None — every frontend page is traceable to at least one requirement.
