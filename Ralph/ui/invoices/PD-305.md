## Where to Find This Feature

This feature lives under the **Invoices** tab. Open any individual invoice from the Invoice List and scroll to the **Attachments** section near the bottom of the page. From there users can upload, preview, and manage PDF and image attachments before the invoice data is transmitted to the external invoicing system.

## User Flows by Acceptance Criterion

---

### AC1 — System supports PDF attachments in FINVOICE format

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice List**.
3. You will see a table of invoices with columns for number, customer, status, and amount.
4. Click the **View** button on any invoice row to open the invoice detail page.
5. Scroll to the bottom of the detail page until you reach the **Attachments** heading.
6. You will see the Attachments panel with a list of current attachments (or the message "No attachments.") and an **+ Upload File** button.

**Confirms requirement:** The Attachments panel exists on every invoice detail page, showing that the system is built to support PDF attachment management for invoicing data in line with FINVOICE expectations.

**UI Status:** ✅ VISIBLE

---

### AC2 — Upload and link files; format, count and size validation

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice List**.
3. Click the **View** button on any invoice row.
4. Scroll to the **Attachments** panel at the bottom.
5. The counter shows how many files have been attached (e.g. "0 / 10 files") and how many kilobytes remain before the 1 MB limit.
6. Click **+ Upload File**. A file browser opens.
7. Select a file. If the file is not a PDF, JPEG, or PNG, you will immediately see the error: "Only PDF/A, JPEG, and PNG files are accepted."
8. If the file would push the total size over 1 MB, you will see: "File too large. Only X KB remaining."
9. If you have already uploaded 10 attachments, the **+ Upload File** button is disabled and the message "Maximum 10 attachments reached." appears.
10. A valid file uploads and the attachment appears in the list with its filename, size in KB, and a **Download** button.

**Confirms requirement:** File type, count (max 10), and total size (max 1 MB before encoding) are all enforced in the UI with clear error messages, matching the FINVOICE attachment rules.

**UI Status:** ✅ VISIBLE

### AC3 — Automatic validation; error notification before transmission

1. Follow the navigation path above to the **Attachments** panel on any invoice detail page.
2. Attempt to upload a file with an unsupported extension (e.g. .docx or .xlsx).
3. You will see an inline error message: "Only PDF/A, JPEG, and PNG files are accepted." — the upload is blocked immediately.
4. Attempt to upload a file that would exceed the remaining storage budget.
5. You will see an inline error message: "File too large. Only X KB remaining." — the upload is blocked.
6. Attempt to upload an 11th file when 10 are already attached.
7. The **+ Upload File** button is greyed out and disabled; the message "Maximum 10 attachments reached." is displayed.

**Confirms requirement:** The system blocks invalid uploads with explicit inline error messages before any data can be transmitted, so the user is always notified before transmission.

**UI Status:** ✅ VISIBLE

### AC4 — SEI security class codes and SHA1 checksums

1. Follow the navigation path above to the **Attachments** panel on any invoice detail page.
2. Before clicking **+ Upload File**, locate the **Security Class** dropdown above the upload button.
3. Click the dropdown. You will see four options: **None**, **SEI01**, **SEI02**, and **SEI03**.
4. Select the desired SEI code (e.g. **SEI01** for restricted documents).
5. Click **+ Upload File** and select a valid PDF or image.
6. After the file uploads, the attachment row appears. Any SEI code set at upload time is shown as a small label next to the filename.
7. The attachment row also displays an **attachment identifier** (the SHA1-derived ID) in a monospace code element next to the filename.

**Confirms requirement:** Users can classify attachments with SEI codes before uploading, and the system displays the SHA1-based attachment identifier on each uploaded file, confirming both security classification and integrity checking are visible.

**UI Status:** ✅ VISIBLE

### AC13 — Use case: upload two PDFs, Base64 encode, generate SHA1, validate, transmit

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice List**.
3. Click **View** on the target invoice row.
4. Scroll to the **Attachments** panel.
5. Select a **Security Class** if required (e.g. **SEI01**).
6. Click **+ Upload File** and select the first PDF (e.g. a customer contract). It must be PDF/A format.
7. The attachment appears in the list with its filename, size, and attachment identifier.
8. Click **+ Upload File** again and select the second PDF (e.g. a waste report). The counter updates.
9. Both attachments are now listed and validated. The FINVOICE XML generated by the backend will include them Base64-encoded with SHA1 identifiers.
10. Use the **↓ FINVOICE XML** button to download and inspect the generated XML to confirm both attachments are present before transmitting.

**Confirms requirement:** The UI covers the full upload and listing flow. Base64 encoding and SHA1 generation happen automatically in the backend and are not separately visible to the user, which is expected for an automated compliance step.

**UI Status:** ⚠️ PARTIAL

The upload, file count, size tracking, error messaging, and security classification steps are all visible. The Base64 encoding and SHA1 generation are backend-only (not shown as explicit steps in the UI), which is intentional and correct — these are automated processes, not user actions. The FINVOICE XML download allows indirect verification.

## Backend-Only Aspects

- **Base64 encoding of attachments** — When a file is uploaded, the backend automatically encodes it in Base64 for FINVOICE compliance. The user does not need to perform or confirm this step; it happens transparently on the server.
- **SHA1 checksum generation** — The backend generates a SHA1 checksum for each uploaded attachment to produce the `AttachmentIdentifier`. The resulting identifier is shown in the UI, but the hash computation itself is a backend-only operation.
- **FINVOICE XML assembly** — The backend constructs the full FINVOICE message including the `AttachmentDetails` block (with `AttachmentIdentifier`, `AttachmentName`, `AttachmentMimeType`, `AttachmentContent`, `AttachmentSecurityClass`) when the invoice is exported. This is not a user-facing step; the user sees the result via the **↓ FINVOICE XML** download.
- **AC5, AC6, AC7** — These are clarification notes in the PRD (terminology and architectural decisions) with no corresponding discrete UI action.
