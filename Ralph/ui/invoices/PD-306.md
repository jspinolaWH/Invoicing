## Where to Find This Feature

This feature lives under the **Invoices** tab. Open any invoice from the Invoice List to reach the detail page, then scroll down to the **Transmission & Integration** panel where the **View Invoice Image (PDF)** button appears (visible only after the invoice has been transmitted to an external operator).

---

## User Flows by Acceptance Criterion

---

### AC1 — Real-time retrieval and display of invoice images from an external system

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice List**.
3. You will see a table of invoices showing Invoice #, Customer, Date, Net, Gross, Status, and Billing Type.
4. Click any invoice row that has a status of **SENT** (rows are clickable).
5. You will land on the invoice detail page. Scroll down past the line items, amount summary, custom text, and attachments sections.
6. You will reach a panel titled **Transmission & Integration**. If the invoice has been transmitted, it carries an external reference number.
7. Click the **View Invoice Image (PDF)** button (teal/cyan coloured).
8. The system sends a live request to the backend, which in turn calls the external invoicing system. Within seconds, a PDF viewer iframe appears directly below the button group on the same page, showing the invoice image.

**Confirms requirement:** The user sees the invoice PDF rendered inside the ERP page without leaving the application, confirming that real-time retrieval and in-ERP display works.

**UI Status:** ✅ VISIBLE

---

### AC2 — User selects an invoice and the image opens in a dedicated window

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice List**.
3. Click on an invoice row to open the detail page.
4. Scroll down to the **Transmission & Integration** panel.
5. Click **View Invoice Image (PDF)**.
6. After loading completes (the button label temporarily changes to "Loading…"), a dedicated PDF viewer panel opens inline on the page. The viewer has a header bar labelled **Invoice PDF** and an X button to close it.
7. The PDF is rendered inside an embedded frame at full width, approximately 500 pixels tall, so users can read invoice details without downloading any file.

**Confirms requirement:** The invoice image opens in an in-page viewing window directly within WasteHero, eliminating the need to switch to another system.

**UI Status:** ✅ VISIBLE

---

### AC3 — Multi-provider compatibility, role-based access, and error handling

1. Click **Invoices** in the top navigation bar.
2. Click **Invoice List** in the left sidebar.
3. Open an invoice and scroll to the **Transmission & Integration** panel.
4. **Error handling:** If the external system is unavailable or the invoice image cannot be retrieved, a red error banner appears immediately below the button group, reading: "The invoice image could not be retrieved. [error detail] Contact your administrator if the problem persists."
5. **Multi-provider compatibility:** The API endpoint `/api/v1/invoices/{id}/image` is provider-agnostic; switching providers requires only a backend configuration change with no UI impact.
6. **Role-based access:** The **View Invoice Image (PDF)** button is currently displayed to all logged-in users without any frontend role check. Role enforcement exists at the backend API level (Spring Security) but is not reflected in the UI — the button does not hide or disable itself based on the current user's role.

**Confirms requirement:** Error messages are visible. Multi-provider support is transparent to the user. Role-based hiding of the image button is absent from the frontend.

**UI Status:** ⚠️ PARTIAL

**What is missing from the UI:** The "View Invoice Image (PDF)" button is shown to all users regardless of role. The requirement states that only authorised users may view invoice images. While the backend enforces this via access control (a 403 would be returned for unauthorised users), the UI does not proactively hide the button or display a "you do not have permission" message — it only shows a generic error after a failed fetch. A proper implementation would check the current user's roles on page load and either hide the button or show a disabled state with an explanatory tooltip.

### AC4 — Accounts receivable not stored in ERP; image retrieved from external system

This criterion is a design statement rather than a user interaction. There is no "accounts receivable" screen anywhere in the application — the ERP manages invoicing data only. The invoice image is always fetched on demand from an external system, never served from a local store.

**Confirms requirement:** The absence of any AR module in the application confirms this design intent. Users cannot access stored invoice images from within ERP data — they can only retrieve them on demand from the external system.

**UI Status:** ✅ VISIBLE

---

### AC5 — Invoice images are viewed exclusively via the integration interface

1. Click **Invoices** in the top navigation bar.
2. Click **Invoice List** in the left sidebar.
3. Open any invoice.
4. The only way to view the invoice as a rendered document is through the **View Invoice Image (PDF)** button in the **Transmission & Integration** panel. There is no "preview invoice" option anywhere else on the detail page.
5. All invoice rendering happens through the external system integration — the ERP itself does not generate or cache PDF copies locally.

**Confirms requirement:** The single entry point for viewing invoice images is the integration panel, confirming that display is exclusively via the integration interface.

**UI Status:** ✅ VISIBLE

---

### AC6 — ERP communicates with external systems via API for smooth image retrieval

This is an infrastructure criterion. From the user's perspective, clicking **View Invoice Image (PDF)** results in the image appearing within seconds with no manual steps. No file downloading, no system switching, and no credentials entry are required.

The frontend calls `/api/v1/invoices/{id}/image` and receives a binary blob, which is converted into an in-page object URL and displayed directly in the embedded iframe.

**Confirms requirement:** Users experience seamless retrieval — clicking one button is all that is required to see the invoice image within the ERP.

**UI Status:** ✅ VISIBLE

---

### AC7 — Only authorised users can view invoice images

Currently, the **View Invoice Image (PDF)** button appears for any user who can open an invoice detail page, regardless of their role or permissions. If the backend denies the request, a generic error message is displayed — but there is no proactive UI gating.

There is no frontend role check (no call to `/api/v1/me/roles` in the invoice detail page or the transmission panel) that would hide or disable this button for unauthorised users.

**Confirms requirement:** This criterion is not fully satisfied in the UI. The button appears universally rather than only for authorised users.

**UI Status:** ❌ NOT VISIBLE

**Why there is no UI surface:** The frontend `InvoiceTransmitPanel` does not import or use `getMyRoles()`. Role-based visibility is enforced only at the API layer. A user with an unauthorised role will see the button, click it, and receive a red error message — rather than never seeing the button at all. This represents a UX gap against the requirement.

### AC8 — Real-time image retrieval triggered by user selecting an invoice

1. Click **Invoices** in the top navigation bar.
2. Click **Invoice List** in the left sidebar.
3. Click an invoice row. The invoice detail page loads.
4. Scroll to the **Transmission & Integration** panel.
5. Click **View Invoice Image (PDF)**. The button label changes to "Loading…" while the request is in flight.
6. The image appears in the inline viewer as soon as the response arrives — no page reload required.

**Confirms requirement:** Each time the user clicks the button, a fresh request is sent to the external system in real time. There is no caching or pre-fetching; the retrieval happens on demand.

**UI Status:** ✅ VISIBLE

---

### AC9 — Seamless integration with external systems via API

From the user's perspective, clicking **View Invoice Image (PDF)** is the only action needed. No login to an external system, no file management, and no additional steps are required. The integration is transparent.

**Confirms requirement:** The integration panel provides a one-click experience for retrieving and displaying the invoice image, confirming seamless API-based integration.

**UI Status:** ✅ VISIBLE

---

### AC10 — Retrieved invoice images displayed in a dedicated in-ERP viewing window

1. Open any transmitted invoice via **Invoices** → **Invoice List** → click a row.
2. Scroll to the **Transmission & Integration** panel and click **View Invoice Image (PDF)**.
3. A bordered container appears below the button row. Its header reads "Invoice PDF" with an X close button.
4. Inside the container, the PDF is rendered in a full-width iframe (500 px tall), allowing the user to scroll through the document within the ERP page itself.
5. To close the viewer, click the **X** button in the panel header.

**Confirms requirement:** The invoice is displayed in a dedicated in-page window within WasteHero — no external tab or application is opened.

**UI Status:** ✅ VISIBLE

---

### AC11 — Consistent invoice tracking — each invoice linked to its image

Each invoice detail page fetches the image for that specific invoice only (the API call uses the invoice's unique ID). Navigating to a different invoice and clicking the same button retrieves that invoice's image from the external system.

**Confirms requirement:** The system correctly maps each invoice in the ERP to its corresponding image in the external system, ensuring consistent tracking.

**UI Status:** ✅ VISIBLE

---

### AC12 — Error handling: clear messages and troubleshooting guidance if image cannot be retrieved

1. Click **Invoices** → **Invoice List** → open an invoice.
2. Scroll to the **Transmission & Integration** panel.
3. Click **View Invoice Image (PDF)**.
4. If the external system is unreachable or returns an error, a red banner appears immediately in the panel with the message: "The invoice image could not be retrieved. [technical error detail] Contact your administrator if the problem persists."
5. The existing PDF viewer (if any was previously loaded) is cleared when a new fetch is attempted.

**Confirms requirement:** The system presents a clear, human-readable error message and directs the user to contact an administrator when the image cannot be retrieved.

**UI Status:** ✅ VISIBLE

## Backend-Only Aspects

**AC4 (AR not stored in ERP)** — This is a pure architectural decision. There is no user-facing screen for it; it is simply the absence of an accounts receivable module in the application. The design is intentional.

**AC6 / AC9 (API connectivity)** — The actual HTTP communication with external invoicing systems happens entirely in the backend service. The frontend only initiates the request by calling the backend endpoint; the backend handles provider-specific API calls, authentication with the external system, and response normalisation.
