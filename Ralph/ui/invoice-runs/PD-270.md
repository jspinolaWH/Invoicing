## Where to Find This Feature

This feature spans the Invoices tab (Invoice Runs) and the Customers & Config tab (Billing Profiles). The lock enforcement and status indicator live on a customer's Billing Profile page; the billing run itself is started and monitored from Invoice Runs.

## User Flows by Acceptance Criterion

---

### AC1 — Customer data can be updated while a billing run is in progress

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice Runs**.
3. Click **New Run**, configure the filters, and click **Start Run** (or navigate to an already-running run in the runs list).
4. While the run is in progress, open a different browser tab or return to the top nav.
5. Click **Customers & Config** in the top navigation bar.
6. In the left sidebar, click **Billing Profiles**.
7. Click any customer row to open that customer's billing profile.
8. You will see the full billing profile form — it is still rendered and accessible while the run is processing.

**Confirms requirement:** The application remains fully navigable and the billing profile form is accessible while a run is in progress; most fields can still be edited and saved.

**UI Status:** ⚠️ PARTIAL

The form is accessible, but the UI does not proactively show a global "run is in progress" notification banner across the app. The lock indicator is only visible on the individual customer's profile page if that specific customer is currently locked.

---

### AC2 — Fields dynamically classified as editable or restricted

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. Click a customer row whose invoice is currently being processed in an active run.
4. At the top of the customer's name heading, you will see an amber badge labelled **In Active Run — Locked**.
5. If you hover over this badge, a tooltip reads: "Billing address and billing group changes are locked while an invoice run is in progress."
6. All other fields on the form remain fully editable.

**Confirms requirement:** A lock badge appears next to the customer name when the backend has flagged that customer as locked. The tooltip explains which fields are restricted.

**UI Status:** ⚠️ PARTIAL

The badge is informational only. The billing address input fields themselves are NOT visually disabled or greyed out — they remain interactive. The restriction is enforced at the API layer (the backend returns an error if the user attempts to save a locked address), but the UI does not pre-emptively disable those inputs or show inline "locked" styling on the individual fields.

---

### AC3 — ERP remains fully operational for all users during a billing run

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice Runs**.
3. Start a new run. Once it is in RUNNING status, navigate freely to any other section (Operations, Invoices, Customers & Config, Master Data, Tools).
4. All pages load and remain fully functional — billing events can be reviewed, invoices browsed, master data maintained.

**Confirms requirement:** No part of the application locks down during an invoice run. All pages and actions remain accessible to all users throughout the run.

**UI Status:** ✅ VISIBLE

---

### AC4 — Non-invoicing customer data can be managed during a run

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. Click any customer row to open the billing profile.
4. You will see fields for Delivery Method, Language Code, Business ID, Invoice Template, Invoicing Mode, Legal Classification, Ledger Code, and e-invoice settings.
5. These fields are not related to the billing address and remain fully editable and saveable during an active invoice run.

**Confirms requirement:** Non-address billing profile data can be edited and saved without restriction while a billing run is running.

**UI Status:** ✅ VISIBLE

---

### AC5 — Billing address cannot be changed during an active run

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. Click a customer whose invoice is being processed.
4. You will see the **In Active Run — Locked** amber badge at the top of the page.
5. Edit any of the billing address fields (Street Address, Postal Code, City, Country Code) and click **Save Billing Profile**.
6. The system will display a red error message at the top of the form with the rejection reason returned by the backend (e.g. "Invoice processing in progress. Address changes cannot be made during this time.").

**Confirms requirement:** Attempting to save a billing address change while the customer is locked results in a visible error message from the backend.

**UI Status:** ⚠️ PARTIAL

The restriction is enforced by the backend (HTTP 409/423) and the error text is surfaced in the UI error banner. However, the address input fields are not visually disabled before the user attempts to save — there is no pre-emptive UI prevention. The user must attempt the save to receive the warning.

---

### AC6 — Billing groups (bundling rules) cannot be modified during an active run

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. Click a customer row to open their billing profile.
4. At the top of the page, click the **Bundling Rules** button (a secondary button just below the page header).
5. You will be taken to the Bundling Rules page for that customer, where you can add, remove, or change bundling rule types.
6. If the customer is currently locked by an active run and you click **Save Bundling Rules**, the backend will reject the request and an error message will be displayed.

**Confirms requirement:** The bundling rules save is blocked by the backend when the customer is locked in an active run. However, the Bundling Rules page itself does not display the lock badge or disable inputs pre-emptively.

**UI Status:** ⚠️ PARTIAL

The Bundling Rules page does not import or display the `CustomerLockedBadge` component. There is no lock indicator or field disabling on this page — the block happens only at the API level with an error shown after an attempted save.

### AC7 — Waste collection schedules editable if change does not affect current period

1. There is no frontend page in this application that manages waste collection schedules.
2. Waste collection schedules are a concept in the WasteHero operational platform (Compass/Bridge domain), not in the invoicing ERP.

**Confirms requirement:** Not applicable — this AC covers a different system. The invoicing ERP has no UI for managing waste collection schedules.

**UI Status:** ❌ NOT VISIBLE

This acceptance criterion refers to waste collection scheduling functionality that lives outside the invoicing ERP. The invoicing system consumes events that result from schedules but does not manage the schedules themselves. No frontend surface exists for this AC within this codebase.

### AC8 — General customer data (contact details, contract terms) can be updated during a run

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. Click any customer row to open the billing profile.
4. You can edit and save: Customer ID Number, Delivery Method, Language Code, Business ID, Invoice Template, Invoicing Mode, Legal Classification, Ledger Code, Invoice Per Project toggle, Allow External Recall toggle, and Parent Customer Number.
5. For contracts, scroll down to the **Contracts** section — you can update the Invoice Template override, Work Site, Start Date, and End Date for each contract and save them individually.
6. All of these save without restriction regardless of whether a billing run is active.

**Confirms requirement:** All non-address billing profile data and contract metadata fields are editable and saveable without restriction during an active billing run.

**UI Status:** ✅ VISIBLE

---

### AC9 — System alerts users when a billing run is active

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. Click any customer row to open their billing profile.
4. If that customer is currently involved in an active invoice run, an amber badge labelled **In Active Run — Locked** appears immediately to the right of the customer name in the page heading.
5. Hovering over the badge shows a tooltip: "Billing address and billing group changes are locked while an invoice run is in progress."

**Confirms requirement:** An in-page amber badge notifies the user that this specific customer is locked due to an active billing run.

**UI Status:** ⚠️ PARTIAL

The notification only appears on the individual customer's billing profile page. There is no app-wide banner or notification that a billing run is currently in progress. Users on other pages (billing events, invoices, etc.) have no visual indication that a run is active. The badge also does not appear on the Bundling Rules page.

---

### AC10 — Warning message shown if a restricted action is attempted

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. Open a customer profile that shows the **In Active Run — Locked** badge.
4. Change any billing address field and click **Save Billing Profile**.
5. A red error banner appears at the top of the Profile tab, displaying the rejection message from the backend.

**Confirms requirement:** Attempting to save restricted data while a customer is locked shows a red error message to the user.

**UI Status:** ⚠️ PARTIAL

The warning is shown reactively after a failed save. There is no proactive pre-flight check that disables the Save button or shows an inline warning before the user clicks Save. The specific message text ("Invoice processing in progress. Address changes cannot be made during this time.") is returned by the backend — the UI displays whatever message the API returns.

---

### AC11 — Large batches (5,000 invoices) processed without system slowdowns

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice Runs**.
3. Click **New Run** and configure a broad filter set that would cover a large customer base.
4. Select **Real Run** and click **Start Run**.
5. You are redirected to the Invoice Run Detail page, which polls every 3 seconds and shows PENDING → RUNNING → COMPLETED status transitions in real time.
6. The detail page shows total invoices processed and total amount once complete.

**Confirms requirement:** The UI provides a real-time polling status view during the run, but there is no UI element that specifically tests or displays performance metrics for large batches. Whether 5,000 invoices can be processed without slowdown is a backend concern.

**UI Status:** ❌ NOT VISIBLE

The performance guarantee for large batches is a backend/infrastructure concern with no dedicated frontend surface. The run detail page shows progress and status, but does not display throughput metrics, progress percentage, or any performance indicator. The non-disruption of other users is also backend-only.

---

### AC12 — All functionalities fully restored once billing run is complete

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice Runs**.
3. Click on a run row whose status is **COMPLETED** or **COMPLETED WITH ERRORS**.
4. Navigate to **Customers & Config** → **Billing Profiles** and open a customer who was previously locked.
5. The **In Active Run — Locked** badge will no longer appear on the customer's heading.
6. All billing address fields can now be edited and saved without restriction.

**Confirms requirement:** Once the billing run finishes, the lock badge disappears and address changes save successfully, confirming that full functionality is restored.

**UI Status:** ✅ VISIBLE

---

### AC13 — All changes during a billing run are logged for audit purposes

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Billing Profiles**.
3. Click any customer row to open their billing profile.
4. Click the **Change History** tab in the tab bar (alongside Profile and Credit History).
5. You will see a table with columns: Field, Old Value, New Value, Changed By, and Changed At.
6. Any edits made to the billing profile — including those made during a billing run — are listed here with a timestamp and the user who made the change.

**Confirms requirement:** The Change History tab provides a full audit log of all billing profile changes, satisfying the audit and compliance logging requirement.

**UI Status:** ✅ VISIBLE

---

### AC14 — Invoice generation remains compliant with contractual and regulatory requirements

1. There is no dedicated UI screen that explicitly certifies or displays regulatory compliance of a billing run.
2. The Invoice Run Detail page shows validation report summaries if validation failures occurred, which partially surfaces compliance issues.
3. Compliance is enforced through classification rules, validation rules, and VAT rate configurations, which can be reviewed under **Customers & Config** → **Classification Rules** / **Validation Rules** and **Master Data** → **VAT Rates**.

**Confirms requirement:** Compliance is enforced by backend validation and configuration, not through a dedicated compliance UI.

**UI Status:** ❌ NOT VISIBLE

This AC is satisfied through backend enforcement (validation rules, VAT rate application, contract terms). The frontend provides the configuration screens for those rules but has no screen that explicitly certifies compliance during a billing run.

## Backend-Only Aspects

- **AC7 (Waste collection schedules):** Waste collection schedule management is outside the scope of this invoicing ERP. The scheduling system lives in the operational domain (Compass/Bridge). This AC cannot be verified in this codebase.
- **AC11 (Large batch performance):** Whether 5,000 invoices can be generated without system slowdown is a backend/infrastructure guarantee. The frontend offers no throughput metrics or performance indicators.
- **AC14 (Regulatory compliance):** Compliance with contractual and regulatory requirements is enforced by backend validation logic (classification rules, VAT configuration, contract date enforcement). The frontend provides configuration UIs for these rules but no run-time compliance dashboard.
