## Where to Find This Feature

This feature is spread across three areas of the app. Validation rules are created and managed under the **Customers & Config** tab. Error detection on individual billing events is visible on the **Billing Events** list (Operations tab) and on each event's detail page. During invoice run setup, validation results are shown on the Simulation Results and Generate Invoice pages (Invoices tab).

## User Flows by Acceptance Criterion

---

### AC1 — All required invoicing data fields are validated before finalising; rules are configurable

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Validation Rules**.
3. You will see a table listing all currently configured validation rules. Each row shows a Rule Code, Type (e.g. MANDATORY FIELD, VAT ACCURACY), whether it is Blocking, whether it is Active, and a Description.
4. These rules are the system's configurable error-checking definitions. Any rule marked **Blocking: Yes** will prevent invoice generation if violated.

**Confirms requirement:** The page proves that the system has a configurable set of validation rules that check required invoicing data fields before invoices are generated.

**UI Status:** ✅ VISIBLE

---

### AC2 — Checks performed during invoice creation; blocking errors stop processing

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Generate Invoice**.
3. Enter a Customer ID, a Billing Period From date, and a Billing Period To date.
4. Click **Preview Invoice**. The system runs a simulation and displays the Invoice Preview panel.
5. If any active blocking validation rules are violated, a red message appears: "[N] blocking validation failure(s) — cannot generate invoice."
6. The **Confirm & Generate** button remains disabled as long as blocking failures exist.
7. To proceed, the user must go back to fix the flagged data (e.g. add a missing billing address or VAT rate), then re-preview until no blocking failures remain.

**Confirms requirement:** The Generate Invoice page actively checks billing data before the user can confirm generation; blocking failures disable the generate button entirely.

**UI Status:** ✅ VISIBLE

---

### AC3 — Automatic error detection when invoice data is created

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. You will see a table of billing events. Each row has a **Validation** column showing a badge: Passed (green), Failed (red), Override (amber), or blank (not yet validated).
4. The stat strip at the top of the page includes a **Validation Failed** tile showing the count of events with failed validation. Clicking it filters the table to show only failed events.
5. To trigger validation on one or more events, tick the checkboxes on the rows you want to check, then click the **Validate (N)** button in the toolbar above the table.
6. A validation report modal opens showing which events passed or failed.

**Confirms requirement:** Billing events display their validation status automatically and the user can trigger validation checks at any time, with results clearly surfaced in the list.

**UI Status:** ✅ VISIBLE

---

### AC4 — Invoice is flagged and users are notified on error

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. In the **Validation** column you will see a red **Failed** badge on any event that did not pass validation.
4. The stat strip tile **Validation Failed** shows the total count; clicking it filters the list to those events.
5. Click **View** on any event with a Failed badge.
6. On the event detail page, scroll down to the **Validation Status** section. You will see a red "Failed" badge and, below it, a list of the specific validation errors (each showing the rule type and a plain-English description in a red notification box).

**Confirms requirement:** Failed events are visually flagged in both the list view and the detail page, and the specific errors are listed so users know exactly what to correct.

**UI Status:** ✅ VISIBLE

---

### AC5 — Companies can define their own error-checking rules

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Validation Rules**.
3. Click **+ Add Rule** in the top-right corner.
4. A modal opens. Fill in the **Rule Type** (choose from the dropdown), a unique **Rule Code**, optional **Config JSON** (to target a specific field), an optional **Description**, and set whether the rule is **Blocking** and **Active**.
5. Click **Save**. The new rule appears in the table immediately.
6. To modify an existing rule, click **Edit** on its row; to remove it, click **Delete**.

**Confirms requirement:** Any user with access can create, edit, and delete validation rules tailored to the company's own invoicing requirements.

**UI Status:** ✅ VISIBLE

---

### AC6 — Common validation rule types (missing address, account numbers, VAT, cost centres)

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Validation Rules**.
3. Click **+ Add Rule**.
4. In the **Rule Type** dropdown you will see the following categories:
   - **MANDATORY FIELD** — covers missing billing address, business ID, and other required fields
   - **PRICE CONSISTENCY** — covers invalid or inconsistent financial data / account numbers
   - **VAT ACCURACY** — covers missing VAT rate or incorrect VAT calculations
   - **CLASSIFICATION** — covers events without proper legal classification
   - **REPORTING DATA COMPLETENESS** — covers missing cost centres or accounting identifiers
   - **QUANTITY THRESHOLD** — covers quantity anomalies

**Confirms requirement:** The rule type options map directly to the common validation categories listed in the requirement (address, accounts, VAT, cost centres).

**UI Status:** ✅ VISIBLE

---

### AC7 — Blocking validation prevents invoice from being finalised

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice Runs**.
3. Select **Simulate First** radio button, configure the date range and other filters, then click **Start**.
4. You are taken to the **Simulation Results** page.
5. If blocking failures exist, a red message is shown: "Cannot start real run: [N] blocking validation failure(s) must be resolved first."
6. The **Commit Real Run** button is disabled until all blocking failures are resolved.
7. Alternatively, on the **Generate Invoice** page (Invoices → Generate Invoice), the **Confirm & Generate** button is also disabled when blocking failures are present.

**Confirms requirement:** The system enforces that invoices cannot be generated or committed when blocking validation failures exist.

**UI Status:** ✅ VISIBLE

---

### AC8 — Users must correct flagged errors before invoice can be processed

1. Click **Operations** in the top navigation bar.
2. In the left sidebar, click **Billing Events**.
3. Find an event with a red **Failed** badge in the Validation column. Click **View** on it.
4. Scroll down to the **Validation Status** section.
5. You will see each individual validation failure listed with its rule type and description.
6. The section explains: "Fix the flagged fields, or override with a documented reason if the failures are acceptable."
7. To fix: click **Edit** at the top of the page, correct the offending fields (e.g. add a missing billing address or accounting account), save, then click **Run Validation** on the detail page to re-check.
8. Alternatively, if the failure is acceptable, click **Override Validation**, enter a mandatory written reason, and click **Confirm Override**. The status changes to "Overridden" (amber badge).

**Confirms requirement:** Users cannot simply ignore validation failures — they must either fix the data or explicitly document a reason for overriding before the event can proceed to invoicing.

**UI Status:** ✅ VISIBLE

---

### AC9 — Summary report of detected errors, categorised

1. On the **Simulation Results** page (reached via Invoices → Invoice Runs → Simulate First → Start), a **Validation Error Summary by Category** table is shown when errors exist.
2. The table has columns: Error Category (e.g. "Missing Mandatory Fields", "Incorrect VAT Calculations", "Reporting Data Incomplete"), Blocking count, Warnings count, and Total.
3. Below the summary, a **Validation Issues** table lists every individual failure with Severity badge (red BLOCKING or amber WARNING), Customer name, Rule Type, and Description.
4. On the **Billing Events** list, selecting events and clicking **Validate** opens a validation report modal that also summarises passed and failed events.

**Confirms requirement:** Users see both a high-level categorised summary and a per-event detail list of all errors, enabling efficient triage and correction.

**UI Status:** ✅ VISIBLE

---

### AC10 — Creation of validation rules

1. Click **Customers & Config** in the top navigation bar.
2. In the left sidebar, click **Validation Rules**.
3. Click **+ Add Rule**.
4. Complete the form (Rule Type, Rule Code, optional Config JSON, optional Description, Blocking toggle, Active toggle).
5. Click **Save**. The rule is saved and immediately listed in the table.

**Confirms requirement:** Users can create new validation rules at any time through a straightforward form without any technical knowledge.

**UI Status:** ✅ VISIBLE
