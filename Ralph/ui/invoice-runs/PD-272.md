## Where to Find This Feature

This feature lives under the **Invoices** tab. The simulation is launched from the **Invoice Runs** page and results appear on a dedicated Simulation Results page reached automatically after running the simulation.

---

## User Flows by Acceptance Criterion

---

### AC1 — Run a simulation without finalising invoices

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice Runs**.
3. You will see a list of past invoice runs. Click the **+ New Run** button in the top right corner.
4. You are now on the Invoice Run page. At the top of the form you will see two radio buttons: **Real Run** and **Simulate First**.
5. Select **Simulate First** by clicking that radio button.
6. Fill in the filter fields as needed (Period From, Period To, Municipality, etc.) to define the scope of the simulation.
7. Click the **Run Simulation** button at the bottom of the form.
8. The system processes the simulation and automatically takes you to the Simulation Results page.

**Confirms requirement:** The page clearly separates simulation from a real run via the "Simulate First" radio option, and the button label reads "Run Simulation" rather than "Start Real Run", confirming no real invoices are created.

**UI Status:** ✅ VISIBLE

---

### AC2 — System applies all invoicing rules during simulation

1. Follow steps 1–7 from AC1 above to run a simulation.
2. On the Simulation Results page you will see summary cards showing **Total Invoices**, **Net Total**, **VAT Total**, and **Gross Total** — these are derived by applying pricing, VAT, and allocation rules to the event data.
3. Scroll down to the **Billed Events by Category** table to see the breakdown of billing events with their net amounts.
4. Scroll further to the **Cost Centre Allocations** table to see how costs have been distributed across cost centres.

**Confirms requirement:** The presence of VAT totals, category breakdowns, and cost-centre allocations on the results page demonstrates that all invoicing rules (billing cycles, pricing models, VAT calculations, cost allocations) were applied during the simulation.

**UI Status:** ✅ VISIBLE

---

### AC3 — Summary report displayed after simulation

1. After running a simulation (AC1 steps 1–7), you land on the **Simulation Results** page automatically.
2. At the top you will see summary cards displaying:
   - **Total Customers** — how many customers were included
   - **Total Invoices** — how many invoices would be generated
   - **Failed Customers** — how many customers had errors
   - **Net Total**, **VAT Total**, and **Gross Total** financial amounts
3. Below the summary cards, if there are validation issues, a **Validation Error Summary by Category** table appears listing error categories (e.g. Missing Mandatory Fields, Incorrect VAT Calculations) with counts of blocking errors and warnings.
4. A **Billed Events by Category** table shows event counts and net amounts per category (e.g. waste collection, rental fees).
5. A **Cost Centre Allocations** table shows how amounts are distributed across cost centres.
6. A **Sample Invoices (first 5)** section provides expandable previews of individual simulated invoices.

**Confirms requirement:** The Simulation Results page provides a comprehensive summary report covering total invoice counts, category breakdowns, VAT calculations, cost-centre allocations, and identified issues.

**UI Status:** ✅ VISIBLE

---

### AC4 — Error detection flags missing or incorrect invoice data

1. After running a simulation (AC1 steps 1–7), scroll down the Simulation Results page.
2. If any validation issues were found, a **Validation Issues** table appears listing each issue with:
   - A coloured **Severity** badge — red for **BLOCKING**, amber for **WARNING**
   - The **Customer** name or ID affected
   - The **Rule Type** (e.g. MANDATORY_FIELD, VAT_ACCURACY, PRICE_CONSISTENCY)
   - A **Description** explaining what is wrong
3. The **Validation Error Summary by Category** table above this groups issues by type so you can see at a glance where the biggest problems are.

**Confirms requirement:** The system flags affected customers with categorised, severity-labelled validation failures covering missing fields, incorrect VAT, and financial data issues.

**UI Status:** ✅ VISIBLE

---

### AC5 — Blocking errors prevent committing the real run

1. After running a simulation, if blocking errors exist, scroll to the bottom of the Simulation Results page.
2. You will see a red error message that reads: "Cannot start real run: [N] blocking validation failure(s) must be resolved first."
3. The **Commit Real Run** button is visibly disabled (greyed out) and cannot be clicked.
4. Once all blocking errors are resolved in the underlying data and the simulation is re-run with no blocking failures, the button becomes active and you can click **Commit Real Run** to proceed.

**Confirms requirement:** The UI explicitly blocks committing a real run when blocking validation failures are present, forcing the user to correct errors first.

**UI Status:** ✅ VISIBLE

---

### AC6 — Filter and analyse simulation results

1. After running a simulation with specific filters (Period From/To, Municipality, Customer Type, Service/Product Code, Billing Frequency, etc.), the results on the Simulation Results page reflect only that filtered scope.
2. The filter settings used are preserved and passed through to the commit action, so the final real run matches the simulation scope exactly.
3. The detailed validation failures table lists each affected customer individually, allowing the user to identify and correct specific records.
4. The category breakdown and cost-centre allocations tables allow the user to analyse results by service type and financial allocation.

**Confirms requirement:** Users can target the simulation to specific segments via run filters, and the results page breaks down data by customer, category, and cost centre for thorough review.

**UI Status:** ✅ VISIBLE

---

### AC7 — Simulation does not modify live invoice data

1. When you select **Simulate First** on the Invoice Run page and click **Run Simulation**, the system calls the simulation API endpoint (separate from the run-creation endpoint).
2. No invoice records are created in the live system — the results exist only on the Simulation Results page in your browser session.
3. If you close or navigate away from the Simulation Results page without clicking **Commit Real Run**, no data is persisted.
4. The Invoice Runs list page shows runs with a **Mode** column: simulations that were committed show a grey **Simulation** badge, while real runs show a green **Real** badge, making the distinction clear.

**Confirms requirement:** The simulation pathway is architecturally separate from the real-run pathway; live invoice data is only created when the user explicitly clicks "Commit Real Run".

**UI Status:** ✅ VISIBLE

---

### AC8 — Audit trail — simulation runs are logged

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice Runs**.
3. The Invoice Runs list displays all past runs including simulation runs.
4. Each row in the table shows the **Mode** column, where simulation runs are labelled with a grey **Simulation** badge, distinguishing them from real runs.
5. Each run entry also shows the run **ID**, **Status**, **Invoices** count, **Total Amount**, **Period**, and **Started** timestamp, providing a traceable record of every simulation executed.

**Confirms requirement:** Simulation runs appear in the Invoice Runs list with their own mode badge and timestamp, providing an audit trail of simulations run prior to final invoicing.

**UI Status:** ✅ VISIBLE
