## Where to Find This Feature

This feature lives under the **Invoices** tab. Users access the simulation through **Invoice Runs** in the left sidebar, where they choose "Simulate First" before running. After simulation completes, they are taken to the **Simulation Results** page which displays a full preview report.

## User Flows by Acceptance Criterion

---

### AC1 — Preview invoices before finalizing via simulation mode

1. Click **Invoices** in the top navigation bar.
2. In the left sidebar, click **Invoice Runs**.
3. You will see a table listing all previous invoice runs with columns for ID, Status, Mode, Invoices, Total Amount, Period, and Started. A **+ New Run** button appears in the top right.
4. Click **+ New Run**.
5. You will land on the "Invoice Run" page. At the top you will see two radio buttons: **Real Run** and **Simulate First**.
6. Click the **Simulate First** radio button to select simulation mode.
7. Fill in optional filter fields (Municipality, Period From/To, Customer Type, etc.) to scope the simulation.
8. Click **Run Simulation**.
9. You will be taken directly to the Simulation Results page, which shows a full preview of what invoices would be generated — without officially issuing any invoice.

**Confirms requirement:** The user can clearly choose simulation mode before running, triggering a preview without finalizing or sending any invoice.

**UI Status:** ✅ VISIBLE

---

### AC2 — Invoice simulation preview showing financial and service-related data

1. Follow steps 1–9 from AC1 to arrive at the **Simulation Results** page.
2. At the top of the page you will see a summary row of cards showing: **Total Customers**, **Total Invoices**, **Failed Customers**, **Net Total**, **VAT Total**, and **Gross Total** — giving an immediate financial overview.
3. Scroll down to see the **Sample Invoices (first 5)** section, where each entry shows the customer name, gross amount, and number of line items. Click any entry to expand it and see the net and gross breakdown.

**Confirms requirement:** The simulation preview displays financial totals (net, VAT, gross), customer count, and sample invoice line items that mirror what the final invoice would contain.

**UI Status:** ✅ VISIBLE

---

### AC3 — Data retrieval from event records, customer profiles, pricing, and accounting

1. Follow steps 1–9 from AC1 to arrive at the **Simulation Results** page.
2. Scroll to the **Billed Events by Category** section. This table lists each event category (e.g. waste collection, transport fees), the count of events, and the net amount — confirming event records have been retrieved and priced.
3. Scroll further to the **Cost Centre Allocations** section. This table lists each cost centre code, description, number of events, and net amount — confirming accounting and cost centre assignments have been applied.
4. The **Sample Invoices** section at the bottom shows per-customer breakdowns including gross amounts, confirming customer profile and pricing data has been applied.

**Confirms requirement:** The simulation report surfaces data from all required sources: event records (category breakdown), pricing configurations (net/gross amounts), customer profiles (per-customer breakdown), and accounting allocations (cost centre table).

**UI Status:** ✅ VISIBLE

---

### AC4 — Mass invoice simulation for large volumes of invoices

1. Follow steps 1–8 from AC1, but on the Invoice Run filter form, leave the filters broad (e.g. no municipality filter, wide date range) to include the full customer base.
2. Click **Run Simulation**.
3. The Simulation Results page will show **Total Customers** and **Total Invoices** cards at the top, confirming that all customers in scope have been processed in a single simulation pass.

**Confirms requirement:** The simulation is not limited to individual customers — it processes all customers matching the filter criteria in a single bulk operation, and the totals card confirms the volume processed.

**UI Status:** ✅ VISIBLE

---

### AC5 — Error detection and validation before sending

1. Follow steps 1–9 from AC1 to arrive at the **Simulation Results** page.
2. If validation issues are found, a **Validation Error Summary by Category** table appears below the summary cards. It groups errors into categories such as "Missing Mandatory Fields", "Incorrect VAT Calculations", "Financial Data Issues", etc., showing counts of Blocking errors and Warnings for each.
3. Below that, a **Validation Issues** table lists each individual issue with its Severity (shown as a red "BLOCKING" or amber "WARNING" badge), the affected Customer name, the Rule Type, and a plain-English Description.
4. At the bottom of the page, if there are any BLOCKING failures, the **Commit Real Run** button is disabled and an error message reads: "Cannot start real run: [n] blocking validation failure(s) must be resolved first."
5. If all issues are Warnings only (no blocking), the **Commit Real Run** button is enabled and the user may proceed.

**Confirms requirement:** The user can clearly see all validation errors categorised and described, and is physically prevented from committing a real run while blocking issues (e.g. missing VAT information, incorrect pricing) remain unresolved.

**UI Status:** ✅ VISIBLE

---

### AC6 — Audit and compliance support — review system logs during simulation phase

1. After a simulation is committed as a real run via **Commit Real Run**, a run detail page opens at Invoice Run #[id].
2. The run detail page shows run metadata including start time, completion time, total invoices, and filters applied.
3. To review per-invoice audit information, scroll to the **Invoices in this Run** section and click any invoice row to open the Invoice Detail page.
4. The Invoice Detail page includes a reporting audit trail section that tracks changes to reporting fields over the invoice's lifecycle.
5. However, simulation-specific audit log entries (i.e. what happened *during* the simulation phase before commit) are not separately surfaced in a dedicated simulation audit view — the audit trail begins from when the real run is committed.

**Confirms requirement:** Partial — invoice-level audit trails are visible on the Invoice Detail page, but there is no dedicated simulation-phase audit log showing changes or checks performed specifically during the simulation run itself.

**UI Status:** ⚠️ PARTIAL

**What is missing:** The simulation results page and the invoice run detail page do not expose a timestamped log of what was checked, in what order, and with what outcomes during the simulation phase. The requirement calls for full traceability during simulation; the current UI shows validation outcomes but not a step-by-step simulation audit trail.
