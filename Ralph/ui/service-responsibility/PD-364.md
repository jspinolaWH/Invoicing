## Where to Find This Feature

This feature lives under the **Tools** tab in the top navigation bar. It is accessed through the **Resp. Change** sidebar item, which opens the Service Responsibility Change page where users can preview and apply retroactive responsibility transfers between customers.

---

## User Flows by Acceptance Criterion

---

### AC1 — Retroactive changes are applied to service records, reports, and invoicing management tools

1. Click **Tools** in the top navigation bar.
2. In the left sidebar, click **Resp. Change**.
3. You will see the "Service Responsibility Change" form with fields for customer numbers, date range, change effective date, new service responsibility, reason, and internal comment.
4. Fill in the **From Customer Number** (the currently responsible customer) and the **To Customer Number** (the new responsible customer).
5. Set the **Change Effective Date** to define when the change takes effect.
6. Select the desired **New Service Responsibility** from the dropdown (Municipality, Private Collector, Contract Holder, Shared, or Owner).
7. Click **Preview Changes**. The system will display a table of all affected billing events, showing Event ID, Date, Product, Quantity, Status, and Fee Amount.
8. Review the preview summary which shows how many events are "In-progress (will move)" versus "Already billed (need copy)".
9. Click **Apply Change**, then confirm in the confirmation dialog.
10. You will see a green "Responsibility Change Applied" panel showing counts for moved events, correction copies created, and excluded originals, along with any newly created event IDs.

**Confirms requirement:** The user can see billing events being reclassified under the new responsibility and the system confirms how many records were updated and how many correction copies were created for re-invoicing.

**UI Status:** ✅ VISIBLE

### AC2 — Users can define a start date and the system identifies and updates all related uninvoiced events

1. Click **Tools** in the top navigation bar.
2. In the left sidebar, click **Resp. Change**.
3. In the form, fill in **From Customer Number** and **To Customer Number**.
4. Enter a **Change Effective Date** — this is the date from which the responsibility change takes effect.
5. Optionally set **Event Date From** and **Event Date To** to further narrow the scope of events.
6. Click **Preview Changes**.
7. The preview table shows all uninvoiced (IN_PROGRESS status) events that fall within the specified date range.
8. Review the event list, then click **Apply Change** and confirm.

**Confirms requirement:** The Change Effective Date field allows users to specify the start date for the change, and the preview clearly separates in-progress (uninvoiced) events from already-billed events, showing the system identifies and updates only the relevant uninvoiced ones.

**UI Status:** ⚠️ PARTIAL

**If PARTIAL — explain what is missing from the UI:** The field is labelled "Change Effective Date" but there is no explicit explanation in the UI of how this date interacts with event date filtering. It is not clearly documented whether this date acts as a filter or simply as metadata stored with the change run. The "Event Date From / Event Date To" filters do the actual scoping — the effective date may only be stored for audit purposes. The distinction between these two date concepts is unclear to a non-technical user.

### AC8 — Users can generate reports based on original or updated responsibility for internal and external audits

1. Click **Tools** in the top navigation bar.
2. In the left sidebar, click **Resp. Change**.
3. Scroll down past the form to the **Responsibility Change History** section at the bottom of the page.
4. Click **Load Report**.
5. A table appears showing all past change runs with columns: Applied At, Applied By, From Customer, To Customer, Previous Responsibility, New Responsibility, Effective Date, Affected (event count), and Reason.
6. For each row, click **View Events** to expand a sub-table showing individual billing events affected by that change run, including the field changed, old value, new value, who changed it, and when.

**Confirms requirement:** The user can see both the original (Previous Responsibility) and updated (New Responsibility) classifications for each change run, providing an audit trail for compliance reporting.

**UI Status:** ⚠️ PARTIAL

**If PARTIAL — explain what is missing from the UI:** The report shows the history of changes and affected events, but there is no ability to export this report or filter it by date range directly from the UI (the API supports filtering by customer number only). There is no dedicated "compliance report" that generates a structured document showing original vs updated responsibility across all customers for a given period — a feature that auditors would typically require.

### AC9 — System logs all retroactive changes with user, timestamps, and events affected

1. Click **Tools** in the top navigation bar.
2. In the left sidebar, click **Resp. Change**.
3. Scroll down to the **Responsibility Change History** section.
4. Click **Load Report**.
5. You will see a table with one row per change run showing: the timestamp ("Applied At"), the user who applied it ("Applied By"), the customer accounts involved, the previous and new responsibility, the effective date, and the count of affected events ("Affected").
6. Click **View Events** on any row to see the detailed audit log entries for each individual billing event modified in that run, including the specific field changed, old value, new value, user, and timestamp.

**Confirms requirement:** The audit log table confirms that the system records who made the change, when it was applied, and exactly which events were affected and how they were modified.

### AC11 — System limits retroactive changes to uninvoiced events only

1. Click **Tools** in the top navigation bar.
2. In the left sidebar, click **Resp. Change**.
3. Fill in the customer numbers, date range, and effective date, then click **Preview Changes**.
4. The preview summary table separates events by status: "In-progress (will move)" shows uninvoiced events that will be directly transferred, while "Already billed (need copy)" shows events that cannot be directly modified.
5. A yellow warning banner appears when there are already-billed events, reading: "Note: X already-billed event(s) will be excluded from the original customer and correction copies will be created for the new customer for re-invoicing."
6. The system proceeds only to move in-progress (uninvoiced) events directly; billed events are handled through correction copies, not direct modification.

**Confirms requirement:** The preview explicitly shows the distinction between uninvoiced events (moved directly) and already-billed events (handled via correction copies), confirming that issued invoices are protected.

**UI Status:** ✅ VISIBLE

### AC13 — Municipality scenario: commercial property responsibility transfer mid-year

1. Click **Tools** in the top navigation bar.
2. In the left sidebar, click **Resp. Change**.
3. Enter the municipality's customer number in **From Customer Number** and the regional waste authority's customer number in **To Customer Number**.
4. Set **Event Date From** to 1 January (the retroactive effective date) and **Event Date To** to the current date to capture all service events since that date.
5. Set **Change Effective Date** to 1 January.
6. Select the appropriate **New Service Responsibility** (e.g., Private Collector or Contract Holder) from the dropdown.
7. Enter a reason such as "New regulations transfer responsibility for commercial properties effective 1 January."
8. Click **Preview Changes** to see all affected billing events.
9. Review the split between in-progress and already-billed events.
10. Click **Apply Change** and confirm.
11. A green success panel confirms the change with counts of moved events, correction copies, and excluded originals.

**Confirms requirement:** The tool directly supports the municipal transfer scenario by allowing date-bounded retroactive reassignment of billing events from one customer to another.

**UI Status:** ✅ VISIBLE

### AC14 — Business customer scenario: correct mistaken TSV responsibility from error date

1. Click **Tools** in the top navigation bar.
2. In the left sidebar, click **Resp. Change**.
3. Enter the business customer's current (incorrectly assigned) customer number in **From Customer Number** and the correct customer number in **To Customer Number** (if it is the same customer, note that both fields are required and must differ — in a same-customer correction scenario, only the service responsibility field changes).
4. Set **Event Date From** to the date the error began.
5. Select the correct classification from **New Service Responsibility** dropdown (e.g., Private Collector instead of Municipality).
6. Enter a reason such as "Correction of mistaken TSV service responsibility assignment."
7. Click **Preview Changes**, review the affected events, then click **Apply Change** and confirm.
8. The system applies the correction only to pending (uninvoiced) events. Events that have already been invoiced are not directly modified, preserving accounting integrity.

**Confirms requirement:** The UI enforces that only uninvoiced events are changed directly, and the preview warns clearly about already-billed events — satisfying the accounting protection requirement.

**UI Status:** ✅ VISIBLE

### AC15 — Seasonal scenario: update seasonal service retroactively for a specific summer period

1. Click **Tools** in the top navigation bar.
2. In the left sidebar, click **Resp. Change**.
3. Enter the customer's account number in both **From Customer Number** and **To Customer Number** fields if the responsibility holder stays the same but the classification changes seasonally.
4. Set **Event Date From** to the start of the agreed summer period and **Event Date To** to the end of the summer period to scope only those months.
5. Select the appropriate updated **New Service Responsibility** for the summer classification.
6. Enter a reason describing the tariff change and seasonal scope.
7. Click **Preview Changes**, confirm the listed events match the summer period, then click **Apply Change** and confirm.

**Confirms requirement:** The date range fields allow the user to scope the retroactive change to exactly the agreed summer period, ensuring only those events are reclassified.

**UI Status:** ⚠️ PARTIAL

**If PARTIAL — explain what is missing from the UI:** There is no explicit "seasonal" or "period" concept in the UI — users must manually enter the correct date range. Additionally, requiring both From and To customer numbers for a same-customer classification update may be confusing (the form may reject identical customer numbers depending on backend validation). There is no dedicated workflow to tie the responsibility change to an operational schedule or compliance report confirming the seasonal period is correctly reflected.
