## Where to Find This Feature

This feature is spread across two areas of the app that are not listed in the regular sidebar navigation — they must be reached by navigating directly to the relevant URLs. The **Billing Threshold Configuration** page allows admins to set the euro limits, and the **Billing Threshold Alerts** page (with its Triggers, Tickets, and Customers tabs) is where back-office staff monitor and act on customers who have exceeded their threshold.

## User Flows by Acceptance Criterion

---

### AC1 — Configure annual euro billing thresholds per service responsibility classification

1. Navigate directly to the **Billing Threshold Configuration** page (the URL is `/admin/billing-threshold-config` — this page is not listed in the sidebar navigation, so you must type or bookmark the address).
2. You will see a table showing any existing threshold configurations with columns for **Service Responsibility**, **Annual Euro Limit**, **Description**, and **Created By**.
3. To add a new threshold, click the **+ Add Threshold** button in the top-right corner.
4. A form appears with a **Service Responsibility** dropdown (options: MUNICIPALITY, PRIVATE_COLLECTOR, CONTRACT_HOLDER, SHARED, OWNER) and an **Annual Euro Limit** field in euros.
5. Optionally fill in the **Description** field.
6. Click **Create** to save the threshold.
7. To modify an existing threshold, click **Edit** on any row. The form re-appears pre-filled. Make your changes and click **Update**.
8. To remove a threshold, click **Remove** on the row and confirm the prompt.

**Confirms requirement:** The admin can create, edit, and delete per-classification euro thresholds, making the limit adjustable.

**UI Status:** ✅ VISIBLE

---

### AC2 — System continuously monitors annual billed amounts and creates a trigger when threshold is exceeded

This criterion is satisfied entirely in the backend. The backend service evaluates billed amounts on an ongoing basis and automatically creates trigger records in the database when a configured threshold is crossed. There is no user-facing screen for initiating this monitoring — it runs without any action required from the user.

**Confirms requirement:** Not directly observable in the UI; the existence of triggers in the Triggers tab of the Alerts page is the evidence that the backend process is working.

**UI Status:** ❌ NOT VISIBLE

[NOT VISIBLE — The automatic monitoring and trigger creation is backend logic only. Users cannot start or stop the monitoring from the UI; they can only view the resulting triggers after the backend has created them.]

---

### AC3 — Filter and search business customers by classification and threshold status

1. Navigate directly to the **Billing Threshold Alerts** page (URL: `/alerts/triggers` — not listed in the sidebar).
2. You will see three tabs at the top of the page: **Triggers**, **Tickets**, and **Customers**. Click the **Customers** tab.
3. You will see a **Threshold Status** dropdown filter with the options: All Customers, Exceeded, Not Exceeded.
4. Select **Exceeded** or **Not Exceeded** from the dropdown.
5. Click **Refresh** to apply the filter.
6. The table below updates to show matching customers with columns: **Customer #**, **Annual Amount (YTD)**, and **Status** (a coloured badge — red for Exceeded, green for Not Exceeded).
7. For exceeded customers, a **View Trigger** button appears in the row, which takes you directly to that customer's trigger detail.

**Confirms requirement:** Users can filter business customers by their threshold status (exceeded or not exceeded) and see their annual billing amount at a glance.

**UI Status:** ✅ VISIBLE

[Note: The Customers tab does not include a free-text search by customer name or a classification dropdown filter — it only filters by exceeded/not exceeded status. Filtering by service responsibility classification on the Customers tab is not present, though the Triggers tab does support classification filtering.]

### AC4 — Convert triggers into tickets for structured case management and follow-up

**Option A — From the Triggers list:**

1. Navigate to the **Billing Threshold Alerts** page (`/alerts/triggers`).
2. The **Triggers** tab is shown by default, listing all triggers with columns: Customer #, Classification, Annual Amount, Threshold, Year, Status, and Audited By.
3. Use the **Classification** dropdown, **Status** dropdown, or **Customer #** text field to filter the trigger list, then click **Search**.
4. On any trigger row with status **OPEN**, a **Convert to Ticket** button appears in the last column.
5. Click **Convert to Ticket** and confirm the dialog box.
6. The trigger row updates its status to **CONVERTED TO TICKET**.

**Option B — From the Trigger Detail page:**

1. On any trigger row in the Triggers tab, click **View** to open the trigger detail.
2. On the detail page, if the trigger is **OPEN**, a **Convert to Ticket** button is shown in the action bar.
3. Click **Convert to Ticket** and confirm the dialog.
4. The status badge at the top of the page changes to **CONVERTED TO TICKET**.

**Option C — View all tickets:**

1. On the Billing Threshold Alerts page, click the **Tickets** tab.
2. You will see a table of all tickets with columns: Ticket ID, Customer #, Classification, Status, Assigned To, and Created At.
3. Click **View Trigger** on any ticket row to navigate back to the originating trigger.

**Confirms requirement:** Users can convert an open trigger to a ticket, and view the resulting ticket list.

**UI Status:** ✅ VISIBLE

---

### AC5 — View customer details, billing history, and service responsibility from a trigger or ticket

1. Navigate to the **Billing Threshold Alerts** page (`/alerts/triggers`).
2. On the **Triggers** tab, click the **View** button on any trigger row to open the Trigger Detail page.
3. The detail page shows two info cards in the top section:
   - **Customer Number** and **Service Responsibility** (the current classification)
   - **Annual Billed Amount** (in red, indicating it has exceeded the limit), **Configured Threshold**, **Exceeded By** (amount and percentage), and **Year**
4. Below the info cards, if a customer record is found, a **Customer Profile** section displays the customer's name, CVR/Tax ID, address, and contact details.
5. Below the customer profile, a **Billing History** section shows a table of the latest billing events for that customer in the trigger year, with columns: Date, Product, Qty, Waste Fee, Transport Fee, Eco Fee, and Status.
6. An **Audit Trail** section at the bottom shows when the trigger was created, by whom, and when/by whom it was reviewed.

**Confirms requirement:** From a trigger, users can see the customer's details, service responsibility classification, annual billed amount vs. threshold, and their billing history for the year.

**UI Status:** ✅ VISIBLE

### AC6 — Update service responsibility or document justification for retaining current classification

**To document a justification (record a review decision):**

1. Open the Trigger Detail page for an **OPEN** trigger (follow steps in AC5).
2. Click **Record Review Decision** in the action bar.
3. A form appears with a **Decision / Justification** text area. Enter the reason (e.g. "Customer reviewed — grounds exist for retaining municipal classification").
4. Click **Save Decision**.
5. The trigger status changes to **REVIEWED** and a green **Review Record** panel appears showing the decision text, the reviewer's name, and the timestamp.

**To update the customer's service responsibility to market-based:**

1. On the Trigger Detail page, click **Update Service Responsibility** in the action bar.
2. You are taken to the **Responsibility Change** page (under the Tools tab) where you can process the service responsibility update for the customer.

**Confirms requirement:** Users can either record a written justification for retaining the existing classification, or navigate directly to the Responsibility Change tool to update it.

**UI Status:** ⚠️ PARTIAL

[PARTIAL — The "Record Review Decision" flow correctly captures a justification and marks the trigger as reviewed. However, the "Update Service Responsibility" button is a navigation link to the general Responsibility Change page — it does not pre-populate the customer number or link the change back to this specific trigger. The handoff between the trigger and the actual classification update is manual.]

---

### AC7 — Full audit trail of trigger events, review decisions, and service responsibility changes

**For trigger and review audit trail:**

1. Open any Trigger Detail page (`/alerts/triggers/:id`).
2. The **Audit Trail** section at the bottom of the page shows: Created at, Created by, Reviewed at (if reviewed), and Reviewed by.
3. The **Review Record** panel (when present) shows the full decision text, the reviewing user's name, and the review timestamp.

**For service responsibility change audit trail:**

- The responsibility change audit is handled on the Responsibility Change page (Tools → Resp. Change), which maintains its own change log. There is no link back from the trigger detail to the specific responsibility change record that was made as a result of reviewing the trigger.

**Confirms requirement:** Trigger creation timestamps and review decisions are visible in the trigger detail page. However, there is no unified audit view that links responsibility change events to the trigger that prompted them.

**UI Status:** ⚠️ PARTIAL

[PARTIAL — The trigger-level audit trail (creation and review) is visible. The service responsibility change audit exists separately on the Responsibility Change page. There is no single combined audit view and no traceability link between a trigger review decision and the subsequent responsibility change record.]

### AC8 — Only authorised personnel can configure thresholds, create triggers, and manage tickets

This criterion is enforced in the backend. The frontend pages (`/admin/billing-threshold-config` and `/alerts/triggers`) do not display any visible role-check, permission warning, or restricted-access indicator in the UI code. Any user who can navigate to the URLs can see and interact with these pages in the current frontend implementation. The access control is enforced by the backend API, which returns errors for unauthorised requests.

**Confirms requirement:** Not verifiable from the UI alone — enforced at the API level.

**UI Status:** ❌ NOT VISIBLE

[NOT VISIBLE — No frontend role guard or permission check is implemented on these pages. The pages are not listed in the sidebar navigation (which provides some obscurity), but there is no explicit UI-level access control. Backend API security enforces the restriction.]

## Backend-Only Aspects

- **AC2 (Continuous monitoring):** The backend automatically evaluates annual billed amounts against configured thresholds and creates trigger records whenever a threshold is crossed. There is intentionally no manual trigger-creation UI because this is an automated background process.
- **AC8 (Role-based access control):** Authentication and authorisation for threshold configuration, trigger management, and ticket operations are enforced by the backend API. The frontend does not implement a visible role check.
